# ADR-004: Postgres row-level security, deferred with triggers

Status: accepted (2026-07-20)
Date: 2026-07-20
Owner: the founder (solo maintainer)
Informs: [threat model](../security/threat-model.md) section 7.6, S-202, S-801, TM-1, TM-2; amends the RLS question [ADR-002](ADR-002-stack.md) left open (line 64)

## Context

ADR-002 carried one security question forward verbatim: "Yard isolation (S-202) is application-layer discipline: membership-scoped default managers plus `get_object_or_404`, enforced by the isolation test suite as a merge gate from the first model. One missed scoped queryset is a cross-yard leak. Postgres row-level security as belt and suspenders is a Phase 2 decision, on the record."

Yard isolation is the product's hardest promise: the two sides of the family must never see or infer each other's posts, members, or media. The committed control is application-layer: scoped default managers, one central authorization guard returning a byte-identical 404, the S-202 isolation suite as a merge gate, and a CI rule that fails the build if a read handler skips the guard. The question is whether Postgres RLS should sit underneath that as a second, database-enforced net.

The decision came from the same structured multi-agent method the earlier ADRs disclose: one research agent steelmanned adoption, one steelmanned deferral, one gathered ecosystem evidence with no position, and a fourth weighed the three. This is one founder running agent research, not a human panel, and the ruling is only as good as the grounding behind it, so the load-bearing claims below are checked against current Django 5.2 and PostgreSQL 18 documentation and the repository files, dated 2026-07-20, and the sources are named where they carry weight.

## Decision

**Defer RLS. Do not adopt it in v1. Build now only the parts whose cost rises with deployed instances; defer the parts whose cost recurs on the maintainer. Reopen on any of four named triggers.**

The decision is not a compromise between the two positions. Each side had one argument the other never defeated, and both are absorbed:

- **The case for RLS is a real, verified gap.** Django's related-object traversal uses `Model._base_manager`, and the docs require base managers not to filter ("it is critical not to filter away any results"). So a relation walk from an in-yard object to a related object bypasses the scoped-default-manager half of the committed control by design. The guard-plus-suite control is therefore enumerative and fail-open: it protects the read paths the suite and the lint know about, and the count of those paths grows with every contributor. That is a genuine hole a database policy would close.
- **The case against RLS is a cost the constitution will not pay, and a leak the control would manufacture.** Backyard's audience predicate (a post is visible if its audience includes a yard the viewer shares, or the viewer is in the post's pod, plus per-field directory visibility that is column-level, plus viewer-relative filtering of reactor lists on bridging posts) is more granular than a row policy expresses cleanly, and the PostgreSQL docs themselves say policies that consult other tables via sub-SELECT are not the best-performing case. Encoding it in policy SQL is a partial second implementation of the audience logic, the exact parallel-authz hazard TM-2 was written to close. The system principals (the Procrastinate worker, the digest builder) need cross-member reads to assemble per-recipient output, which pushes toward a `BYPASSRLS` path that recreates the unguarded-batch hole. And the only context mechanism available to an app that connects as one Postgres role is a session GUC set per request, a documented source of a stale-context cross-yard leak: Nile's production writeup (thenile.dev, 2022) shipped exactly this bug, where thread-local tenant context leaked across reused connections and requests were authorized as the previous request's user; imfeld.dev (2023) records that `SET`/`SET LOCAL` under PgBouncer statement pooling "will likely return rows for the wrong users"; and PostgreSQL scopes `SET LOCAL`/`set_config(..., true)` to the transaction, while Django runs in autocommit unless `ATOMIC_REQUESTS` is on and its native connection pool (`OPTIONS pool=True`) reuses connections. The honest weight of these two sub-points: neither is strictly unavoidable. A digest builder could run in-policy with per-recipient transaction-local context instead of `BYPASSRLS`, and the stale-context leak is exactly what the full-kit rule below (transaction-local `set_config`, no session `SET`, pooling banned) exists to prevent. So both reduce to operational cost and maturity, not impossibility. That is the point: the cost argument alone is sufficient, and it is what carries the decision, so a future re-litigation cannot rescue adoption by knocking down the avoidable planks. What is decisive against adopting now is the recurring cost against the constitution, not a claim that RLS cannot be made correct.

The judge's dividing line is cost-structure, not comfort: **everything whose price rises with deployed-instance count ships now; everything whose price recurs on the solo maintainer defers.** Scored against the constitution (boring-to-operate and five-year-archive at 1.5x, [principles](../principles.md) 7 and 8), with the S-202 suite and the mandatory-guard rule committed but never yet given a chance to fail, adopt-now buys a contingent catch at a certain recurring price. Reject-permanently is equally wrong: the unenumerated-read-path failure is real, Medium, and rising with contributor-years on a public repo, and the option to adopt later is cheap to keep open now and expensive to reopen after code exists.

## Build now (because deferring these gets expensive, not cheaper)

None of these is in the repository yet; all four land at the start of wave 1, listed in the [wave plan](../wave-plan.md)'s wave 1 as the "ADR-004 database foundation". Item 1 is also threat row TS-PG-1; items 2 to 4 are schema and architecture rules that become wave 1 acceptance tests:

1. **The database role split (threat row TS-PG-1).** Today the app connects and migrates as a Postgres superuser (the postgres image grants `SUPERUSER` to the created `POSTGRES_USER`), which RLS can never bind to and which gives an app-process compromise the whole cluster. Create `backyard_migrator` (owns the schema, runs migrations) and `backyard_app` (`LOGIN NOSUPERUSER NOBYPASSRLS NOCREATEDB NOCREATEROLE`, table privileges via `ALTER DEFAULT PRIVILEGES`). web and worker connect only as `backyard_app`. This is about 20 lines of init SQL, free at zero instances and an unattended migration on every family's box later. It is correct hardening on its own (the app must not run DDL-capable at runtime for this adversary set) and it is the topology RLS would otherwise need retrofitted. CI asserts the app-role connection reports `is_superuser = off` and that `CREATE TABLE` fails, so nobody later re-points at the superuser and silently restores the un-bindable topology.
2. **A denormalized, indexed `yard_id` on single-yard tables, with a database-level same-yard integrity constraint on cross-row references.** Where a single-yard row references another, a composite foreign key against a `UNIQUE (id, yard_id)` makes a cross-yard reference unrepresentable. This directly blunts the `_base_manager` traversal gap without a second authorization engine: a relation walk from an in-yard anchor to a single-yard child cannot cross a yard. The honest boundary of this control: it covers only tables that carry one `yard_id`. A bridging-pod post whose audience spans both yards has no single `yard_id`, and the audience-join, digest-assembly, and reaction rows hanging off multi-yard posts are exactly where the S-202 hardest cases live. Those tables get no composite-FK guarantee; they stay enforced by the single audience module (item 3) and the isolation matrix (item 4), and this ADR names them explicitly so the schema work knows which tables the DB-level guarantee does and does not cover. It also keeps a coarse RLS policy one migration away for the single-yard tables if a trigger fires.
3. **One audience-resolution module, single-sourced.** The feed, the object guard, search and autocomplete, reactor and commenter lists, the digest builder, and the token surface all consume the same module. The unscoped manager is renamed (`all_objects`) with a CI-enforced allowlist of the few call sites permitted to use it (the revocation handler, the digest builder, the admin plane), and a CI grep bans raw SQL (`connection.cursor`, `.raw(`, `.extra()`) outside that allowlist. This is the structural answer to "one implementation, not two" that RLS would have violated.
4. **The S-202 matrix generated from the model registry, with relation-traversal fixtures.** A new model without an isolation fixture fails the build, so the enumerative control stops depending on a human remembering to add a case; and the fixtures explicitly render objects reachable by relation traversal from an in-yard anchor, because Django documents that this path uses `_base_manager` and bypasses scoped managers.

**Implementation status (2026-07-22, S-723 — closing the rot the Phase-2 retro flagged.)** Item 1 shipped (the role split, with the non-vacuous CI compose-probe). Item 3 shipped as the single audience module (`core/scoping`) that the feed, guard, digest, reactions, and token surfaces all consume, plus the `check_digest_confinement` drift-guard; the ADR's *named* "scoped default managers" / `all_objects` rename did not materialize (reads are scoped by explicit `scoping` functions on the default manager) and stays a retro-tracked residual. **Item 4's registry guard now ships** as `test_isolation_registry.py`: it enumerates the model registry and fails the build unless every core model is consciously classified as an S-202 isolation-covered read surface or exempt-with-a-reason — so a new model can no longer silently skip yard isolation — and it self-tests non-vacuously from both sides. Item 2 (the denormalized `yard_id` + composite-FK) remains deferred; it is the largest residual and is tracked in the Phase-2 retro, not silently assumed to exist.

## Deferred (because the cost recurs on the maintainer)

RLS policies themselves, `FORCE ROW LEVEL SECURITY`, the transaction-local GUC middleware that carries member context to Postgres, the non-owner test harness, and the per-policy completeness tests. The evidence, named so it is checkable, is what makes this the right call for a solo maintainer:

- **Tooling is thin and young.** The only actively maintained dedicated package is `django-rls` (kdpisda), which reached v2.0.0 on 2026-07-13 but has one maintainer, about 90 GitHub stars, and no PgBouncer or connection-pooling guidance in its docs, the pitfall every production account leads with. The mature, maintained multi-tenancy packages are all app-layer, not RLS: `django-scopes` (by pretix's author) and `django-multitenant` (Citus). Django ships no first-party RLS support, so policies otherwise ride hand-written `RunSQL` migrations invisible to the autodetector.
- **No comparable self-hosted Django product uses it for isolation.** A GitHub code search (2026-07-20) found zero `ROW LEVEL SECURITY` hits in zulip, sentry, or pretix (pretix instead uses `django_scopes` across 208 files), and one hit in PostHog that is a test fixture whose own comment notes a superuser bypasses even `FORCE`.
- **Production RLS users keep their app-layer filters anyway.** The Supabase performance benchmarks (discussion #14576) show membership-shaped policies going from 171ms to 9ms only when an explicit app-layer filter is added on top of the policy: RLS is run as a second net under app-layer scoping, never as a replacement for it. Which is the opposite of the "delete the app-layer control" case for adopting RLS here.

## Reopening triggers

Deferral is not rejection. Any one of these reopens the decision, and checking them is a named owner's job (the wave retro), not a background hope:

- **T1, first isolation escape.** Any cross-yard read that reaches `main` or a deployed instance that the S-202 suite plus the mandatory-guard rule should have caught. One escape falsifies "a single enforced perimeter suffices"; reopen immediately, before the fix wave closes.
- **T2, a DB access path outside the single guard.** Raw SQL or `connection.cursor` in product code, a second service reading Postgres directly (reporting, an in-process SMTP sink doing direct reads), or widening the Django admin beyond the network-gated instance-admin plane.
- **T3, the audience model materializes.** If audience resolution stabilizes into a materialized per-(member, post) visibility table maintained by the same resolver, RLS becomes a one-`EXISTS` mirror of the single implementation, the TM-2 second-engine objection dissolves, and Phase 2 should adopt it.
- **T4, the exposure surface outgrows the founding instance.** Sustained external drive-by PRs adding models or read paths, or deployed families beyond a handful (the threat model's own revisit bar). The enumerative control assumes one maintainer reviews every read path; that assumption weakens with every contributor-year.

## The full-kit rule if RLS is ever adopted

Recorded now so a future adoption cannot ship a half-measure that reads as protection while being vacuous: if RLS is adopted, it ships only as `FORCE ROW LEVEL SECURITY` (owners bypass plain `ENABLE`, so without `FORCE` the policies are vacuous on the owning role), plus a non-owner test harness that exercises the policies as `backyard_app`, plus a completeness test reading `pg_class.relrowsecurity` and `relforcerowsecurity` so a table without a policy fails the build, plus transaction-local `set_config` for member context. Two clauses that a naive adoption would miss and that turn RLS vacuous or fail-open: every SQL view over a policied table is created `WITH (security_invoker = true)` (PG15+), because a plain view runs with the view owner's privileges and silently bypasses the underlying policy, and the completeness test never sees it; and every policy fails closed on missing context, so an unset `current_setting('app.member', true)` yields zero rows, never an error-path fallback or a wide read. Never `ENABLE`-and-hope, never session-level `SET` (it leaks across pooled connections, the stale-context leak named above), and Django's connection-pool option stays banned while any session-GUC scheme exists.

## Consequences

The role split and the schema decisions (denormalized `yard_id`, same-yard integrity, single audience module) land in wave 1 as the foundation everything else builds on, and they are correct independent of the RLS question. The residual, carried openly in threat-model section 7.6: until a trigger fires, yard isolation rides on one enforced perimeter with no database net beneath it, and an unenumerated read path is a live cross-yard leak caught by a human noticing rather than a database refusing. The build-now items shrink that surface; they do not close it. The decision to revisit is a standing item on every wave retro, which is the one place this deferral can silently rot into permanence if it is not checked.
