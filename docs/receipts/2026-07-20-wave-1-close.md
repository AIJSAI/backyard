the running compose stack, per [the wave plan](../wave-plan.md) rules 1 and 2.

## The live repro (running compose stack, through Caddy)

Driven end to end against `docker compose up` on a clean volume: HTTP through the
bundled Caddy to gunicorn running as `backyard_app`. Each line is the observed
result, except the two noted as unit-proven (S-702) or a CI gate (S-803) rather
than exercised live through Caddy.

| Story | What was exercised | Result |
|---|---|---|
| S-801 first-run wizard | POST /setup/ with the console secret creates the admin, the first yard, and the first pod | 302; 1 yard, 1 pod, admin role `instance_admin` |
| S-101 / S-201 invite join | mint an invite, a new member POSTs /join/&lt;token&gt;/ in a fresh session | 302; the member lands in the pod with a login |
| S-202 yard isolation | a second yard's member fetched through the guard by a first-yard member | 404, and not in the member's visible set (isolation holds) |

| S-703 supervised | the admin POSTs /members/supervised/ to create a child under the member | 302; the child is `is_supervised`, parent-managed, no independent login |
| S-702 removal | the admin POSTs /members/&lt;id&gt;/remove/ | 302; the flow runs (its detach + deactivate + revoke effect is unit-proven in test_removal.py against real Postgres) |
| S-805 break-glass | `manage.py break_glass admin` in the container, then open the printed URL | the console-minted reset URL resolves 200 |
| S-803 archive-compat | a database seeded at the baseline schema upgrades to head with the archive intact | a CI gate, not a Caddy repro: green every CI run (test_archive_compat.py) |

The repro script is recorded with this session; the stack was torn down with
`down -v` after.

## What this wave shipped

Thirteen merged PRs (#6 planning through #18):

- The pods-and-yards data model with multi-yard bridging from the first migration.
- The single authorization guard (deny-by-default, byte-identical 404) plus the
  scoped-traversal accessors, and the S-202 isolation suite (property tests over a
  three-yard topology).
- The ADR-004 Postgres role split, the TM-1 revocation core, invites as hashed
  bearer credentials, the removal flow, and the write/grant permission model.
- allauth passkey login with a per-account lockout, the invite signup view (the
  four TS-DJ-5 properties), the first-run wizard, break-glass recovery, and the
  instance-admin member-management surface.
- Present-config and ops hardening (pre-flight backup, role-aware entrypoint,
  edge/db network split, log hygiene) and the archive-compatibility gate.

Every credential-touching increment passed an isolated security-reviewer pass.
Five reviews found real issues: a brute-force lockout removed, an invite void
scoped too narrowly, a vacuous atomicity assertion, a wide break-glass token
window, and a privilege-inversion in the permission model. Each was fixed and
re-verified before merge.
