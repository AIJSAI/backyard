per wave-plan rule 5): S-000, S-203, S-204, S-205, S-302, S-303, S-304, S-305,

One story stays at `built`, on purpose: **S-301**. Its card title, description, and
tracking-parameter strip are tested, but its acceptance also lists a preview image,
and the image is deferred to wave 3. Rendering a remote image means either hotlinking
it (the tracking-beacon and IP-disclosure leak threat-model TS-PP-6 forbids) or
re-hosting it, and re-hosting needs the media store that wave 3 builds. The image
cannot ship before the media pipeline exists, so claiming S-301 fully tested now
would be a counterfactual. The card degrades to a bare link in the meantime.

## Security reviews

Seven isolated security reviews ran this wave, one per increment, each in its own
worktree. Six increments came back sound; the low and medium hardening items they
surfaced were all folded before merge (a comment-thread render cap, a reactor-query
cap, a delete-comment method-order fix, an ad-hoc-pod leave restriction, a profile
value-object that no longer carries the raw member).

The link-preview fetcher was the exception and the reason ADR-002 gave S-301 its own
review. It returned three real high-severity findings, all confirmed against the
installed Python and all fixed before merge:

- An IPv6 address that embeds an internal IPv4 (the NAT64 well-known prefix, the
  IPv4-compatible and SIIT forms) passed `is_global` on Python 3.13, so a
  DNS-controlling attacker could have reached the cloud-metadata endpoint on a NAT64
  network. The address check now decodes every embedded IPv4 and re-checks it.
- The fetch ran inside the request transaction with no total deadline, so a slow
  trickle could hold a database connection open on attacker time. The fetch now has
  an eight-second wall-clock budget and runs outside the request transaction.
- A URL or an og:image longer than its column would have raised a database error and
  lost the member's post. Both are now length-guarded, failing closed to no card.

The residual, stated plainly: the link fetch still runs synchronously in the web
container rather than the worker's segmented network (TS-CO-4), because the worker
lands in wave 3. It is bounded by the deadline and size caps and no longer holds the
request transaction, but a member-supplied URL still causes a server-side fetch on
web. This moves to the worker in wave 3.

## Founder decisions for the wave boundary

Accumulated during the wave, batched here rather than raised mid-build:

- **htmx is not yet used.** The whole wave-2 surface is server-rendered plain forms
  that post and redirect. This is the ADR-002 revisit trigger recorded honestly: the
  composer, the picker, and the feed did not need htmx to be correct, so htmx
  progressive enhancement (inline reactions, no-reload compose) is deferred rather
  than forced. No friction to report; the observation is the absence of need so far.
- **The composer does not remember the last audience choice.** S-203's acceptance
  offers this as optional (its hardening states the last choice may be offered but is
  never the committed default). The committed default is always the poster's own pod,
  which is the security-relevant behavior and is tested. Remembering the last choice
  is a deferred nicety.
- **Two accepted security postures**, both from reviews: the feed advances the
  unread-boundary marker on a GET (safe under the SameSite cookie policy, and its
  worst case is a cosmetic marker), and the write-path views are not rate-limited
  (matching the invite-only, author-only posture with body caps). Both are recorded
  as accepted rather than silently carried.
