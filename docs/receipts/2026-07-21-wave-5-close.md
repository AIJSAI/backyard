# Wave 5 receipt: PWA and the elder token path, CLOSED

Date: 2026-07-21. Main at the W5-4 merge. Wave 5 is the last wave, and unlike
waves 3 and 4 it has no founder or provider dependency, so it closes here: every
wave-5 story is `tested`, the exit drill ran live, and the one ADR-002 gate is
measured for the containers that exist.

## What landed: four increments, PRs #42 through #44 plus this drill

Each merged through the armed protection on green CI, with an isolated security
review folded before merge and a scripted live repro on the running compose
stack through Caddy.

1. **#42 the elder token surface (S-102, S-601, S-602).** The product's core
   bet: a low-tech elder opens one handed-over link and sees their family, no
   login, nothing to install. The URL exchanges for an httpOnly session and
   redirects to a clean surface; the capability ceiling is structural (the
   elder views are the only ones that accept an elder session). The review
   caught a real HIGH: `cycle_key()` preserved a co-located login through the
   exchange, so on a shared family tablet the read-and-react elder link could
   reach full data export. Fixed with `flush()`, closed in both directions,
   regression-tested.
2. **#43 provisioning with QR and regenerate (S-104).** The family tech-helper
   generates the link from the roster: the page shows the exact grant before
   minting, states the ceiling, and ends with the link and an inline printable
   QR. Review folded a HIGH (the master-token page now carries the no-store
   header set) and a refresh footgun (a one-time intent nonce, so a browser
   refresh no longer silently kills the just-handed-over link).
3. **#44 the PWA (S-103).** Manifest, deterministic icons, and a service worker
   minimal by design (a fetch handler for installability, no precache, no
   cache). Review folded a MEDIUM: the digest surface shared the base template
   and was planting a root-scope worker on intermittent recipients; gated on
   authentication so every token surface stays worker-free.
4. **W5-4 the wave-exit drill (this PR).** The full revocation-completeness
   drill plus the WCAG checks.

## The exit gate, proven

**Revocation completeness, live through Caddy.** One member held every
credential class at once: a master token, a live elder session, a logged-in web
session, a digest read link, a signed media URL, and a reply-by-email address.
All six were live before. After a single total regenerate, all six were dead on
their next request: the master token, elder session, and digest link each 404,
the web session and media URL each bounce to login, and the reply address
bounces. The unit drill (`test_revocation_drill.py`) holds the same six classes
against both regenerate and the S-702 removal flow, and pins its coverage to the
revocation registry length so a future credential class cannot be added without
extending the drill.

**WCAG: the contrast and structure checks the unit suite can compute (elder
view).** Computed from the rendered colors: body text is 17.4:1 against its
background, past the 4.5:1 AA contrast bar (1.4.3); interactive controls declare
48px tap targets, exceeding the WCAG 2.2 AA 24px minimum (2.5.8); the view is one
column with one obvious way back and no links off the surface. This is three
computed criteria, not the ~50 of full AA: keyboard operability, focus
visibility, and the rest need the browser audit (Playwright plus axe, the
ADR-002 E2E path), which stays post-close.

## The ADR-002 gate: RAM footprint, measured for the containers that exist

The running three-container stack, warm, at steady state: caddy 17 MiB,
postgres 116 MiB, web (gunicorn) 166 MiB, about 299 MiB total, under the
low end of the 0.7 to 1.2 GB estimate. That is expected: the estimate is for
four containers and the fourth, the Procrastinate worker, does not exist yet
(it is the same blocked item as wave 4's W4-B3). The four-container total is
therefore not yet measurable and is not claimed; the honest number is the
three-container footprint above.

## Phase 2 is NOT closed

Wave 5 closes; Phase 2 does not. Two waves remain measurement-gated on decisions
outside the code:

- **Wave 4** cannot flip its stories to `tested` until the chosen email
  provider's delivery-and-bounce matrix is measured (W4-B1), plus the live
  mailbox round-trip (W4-B2). Both need a founder decision and an external
  account.
- **Wave 3** cannot close until ffmpeg transcode latency and RAM are measured on
  target hardware, not the dev machine (the N100-class gate).

Phase 2's exit is every v1 story at `tested`. Waves 1, 2, and 5 are there; waves
3 and 4 are built and proven live but honestly held at `built` until their
measurements run. The goal stays open.
