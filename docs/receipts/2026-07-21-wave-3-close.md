## ADR-002 carried-forward gates — MEASURED

**ffmpeg transcode latency on target hardware** (S-402's "minutes at most"). Run
on a real 2-vCPU Ubicloud runner (`ubicloud-standard-2`), software libx264, no
hardware acceleration — the conservative floor for a self-host box, taken with
the committed production transcode via `manage.py measure_transcode`. Run log:
GitHub Actions `29873024812` (`.github/workflows/measure-transcode.yml`).

| source (1080p) | wall time | vs realtime |
|---|---|---|
| H.264, 30s | 7.7s | 3.9× |
| HEVC, 30s | 8.4s | 3.6× |
| H.264, 60s | 12.9s | 4.6× |
| HEVC, 60s | **14.9s** | 4.0× |

Worst case a 60s clip transcodes in **14.9s**. Honest note: the runner is a
2-vCPU slice of a fast Ryzen 9 7950X3D host, so per-core it beats an N100; even
at 3–4× slower it stays under a minute, well inside the minutes-at-most bar. The
committed encoder is libx264; `BACKYARD_FFMPEG_VCODEC=h264_qsv` moves it onto an
Intel box's Quick Sync (the runbook documents re-measuring on the deployed box).

**pillow-heif decision** — DECIDED: rely on the composer's client-side Canvas
conversion (HEIC → JPEG in browsers that can decode it), and reject raw HEIC that
reaches the server undecoded, rather than shipping the pillow-heif dependency in
v1. Recorded in `core/media.py`. Revisit only if the seed pod hits raw-HEIC
rejections in practice.

## Live repros on the running compose stack

**S-401 photo ingest + client-side resize.** EXIF/GPS stripped and orientation
baked at ingest (server-side, every path — `test_media`, merged #30, live). The
client-side resize (S-401 acceptance 1) proven live in real Chromium against the
composer page served through Caddy: a **4000×3000 / 1.44 MB** photo became
**2048×1536 / 213 KB** JPEG before upload (6.7× smaller). Progressive
enhancement: the server re-encodes and strips regardless, so it is never a
security dependency.

**S-402 video.** Real Procrastinate worker transcoded a real uploaded MP4 (with
an injected location atom) to a 720p H.264 rendition + poster: `status=done`,
files present, stored source location-clean. Served through Caddy: in-yard member
gets `200 video/mp4` (rendition) + `200 image/jpeg` (poster); cross-yard member
`404`; anonymous `302` (login required).

**S-403 access-checked serving.** Every derivative served only through the one
access-checked path; cross-yard member, unknown token, pending/failed video, and
purged file all collapse to a byte-identical 404. Isolation matrix covers every
derivative type — photo original + thumbnail, video rendition + poster
(`test_media`, `test_video` cross-yard-404-for-both-tokens).

**Deletion beats expiry.** Delete purges every stored file (image, thumbnail,
video source, rendition) on commit and the serve path 404s immediately
(`test_delete_post_purges_its_photos`, `test_delete_post_purges_all_video_files`,
`test_deleted_media_404s`).

**S-704 / S-802 backup + restore, encrypted path + security replay.** The
encrypted round-trip run live: `backup_instance` (215040 B) → `age` encrypt
(215288 B) → `age` decrypt (byte-identical) → `pg_restore` into a migrator-owned
scratch DB → the canary member survived (count=1, "Canary"). The security replay
— a crafted archive with a `secret_key` member escaping the media subtree is
rejected ("unexpected member in media archive") — runs in the gate
(`test_restore_rejects_a_media_tar_that_escapes_the_media_subtree`,
`test_restore_promotes_only_the_media_subtree`, the #47 HIGH fix).
