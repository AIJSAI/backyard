# Threat model

Status: sections 1 to 6 are the Phase 1 design model, written before any code. Section 7 is the Phase 2 stack-specific pass (added 2026-07-20, after ADR-002 chose the stack), and it reasons about the actual scaffold in the repository. This model covers the design in stories.yaml and principles.md, and now the committed configuration; it exists to make the security-relevant decisions before they get locked into code, and to hand the story loop a set of acceptance criteria that prove the promises instead of asserting them.

Method: for each of the seven trust boundaries below, a multi-agent analysis pass drafted threats, then a second adversarial model pass reworked each boundary from two angles. One pass pruned threats that are not material for this deployment class. One pass hunted for the gaps the draft missed. Section 7 used the same shape, grounding library-behavior claims against current documentation and configuration claims against the audited files, cited to the line where the claim is load-bearing. Sections 1 to 6 said plainly that no human security review had happened when they were written; section 7 got a second adversarial pass from the security-reviewer agent before merge. That is still a model pass, and no human security review has happened on any section. Everything about the future app is a design commitment, not a fact about a running system.

Scope, as of 2026-07-20: sections 1 to 6 hold the design-level model and deliberately leave stack-specific threats (parser behaviors, framework defaults, container hardening) to section 7, which is now written. The forwardable-token trade-off the README already promised is recorded in ADR-003, referenced throughout and not re-argued here. The Postgres row-level-security question ADR-002 carried forward is decided in ADR-004, summarized in section 7.6.

Deployment class, held constant across every boundary: one hobbyist-run, self-hosted instance on the public internet, 25 to 60 members, a public repo from day one, elders on long-lived account-less token links, a weekly email digest out with reply-by-email in.

## 1. Who we defend against

We calibrate to the people who actually turn up in an extended family, not to a security conference threat list. Naming them keeps the mitigations honest and cheap.

Realistic adversaries we design for:

- The nosy relative who wants to see the other side of the family, or a contact field they were not given.
- The ex-partner or removed member after a divorce or a family rupture, especially where children are involved.
- The stranger holding a forwarded link: the church friend Nana sent the digest to, the person downstream of a group text.
- The opportunistic internet scanner harvesting the GitHub firehose for secrets and spraying every fresh VPS for open ports and default credentials.
- The curious teen with a legitimate account, browser dev tools, and a payload copied from a blog post.
- The compromised elder mailbox: credential-stuffed webmail, a shared household account, a provider-recycled address.

Who we explicitly do NOT design for:

- Nation-states and targeted professional attackers. If someone with real resources targets a specific family, self-hosted hobbyist software is not the control that saves them.
- A malicious instance admin, beyond documenting honestly what an admin can do. Self-hosting puts the database in the operator's hands. E2EE is a non-goal (it breaks the elder surfaces), so the operator's power is a disclosure problem, not a technical one. See T-OP-G1 and section 5.

## 2. Systemic requirements (TM-1 to TM-10)

This is the most important section. The gap-finders on different boundaries kept surfacing the same blind spots from different angles. Rather than repeat a fix in seven places, we bind it once here. These ten requirements are the load-bearing output of the whole exercise. Each names the threats and gaps it answers and the stories whose acceptance criteria will carry the proof.

### TM-1. One credential registry, one revocation handler, one completeness test

Every bearer capability the system mints is entered in a single registry: master token links, per-digest tokens, signed media URLs, reply-by-email addresses, invite links, and sessions. Membership-lifecycle transitions all fire the same revocation handler over that registry: removal, voluntary leave, a pod leaving a yard, death, and any regeneration. Removal is one atomic act, not a checklist a tired admin walks by hand. The handler also runs a custody step: it enumerates supervised-account credentials and elder tokens the departing member manages, forces rotation, and reassigns custody before it completes. A revocation-completeness test asserts that after each transition the old token, an old session, an old digest link, an old signed media URL, and an inbound reply address each return 404 or bounce on the next request. New capability types cannot ship without registering here.

Answers: T-TOKEN-5, T-TOKEN-6, T-YARD-7, credential-custody gap, T-MINOR-1, removed-managing-parent gap, T-EMAIL member-lifecycle gaps, T-REMOVE-1, T-REMOVE-2, membership-shrink gap, invite-reentry gap. Binds: S-702, S-104, S-205, S-403, S-501, S-502, S-701.

### TM-2. The digest runs through the same authorization engine as the web feed

The weekly digest is the highest-volume content-egress channel and the one with no recall, no access log, and no 404. It must not be a second, parallel implementation of audience logic in a batch job. The digest is built per recipient, through the same audience-resolution code path as the feed, and evaluated against live state at send time. Deleted posts, narrowed audiences, and removed members never ship. Per-post audience is honored, so an ad-hoc cousins-pod post never lands in Nana's digest. Multi-yard members get separate per-yard emails, so no single forwardable artifact fuses two yards. The S-202 isolation test perimeter extends to digest rendering, including the upcoming-dates section and every signed media URL.

Answers: T-YARD-9, digest-second-authz gaps (yard, email), per-post-audience-in-digest gap, deletion-not-propagated gap, bridging-digest-bundles-two-yards gap, T-TOKEN-2. Binds: S-501, S-202, S-204, S-903, S-403, S-302.

### TM-3. The audience picker never widens scope silently

The composer defaults to the narrowest scope, the poster's own pod, on every new post. The "remember last choice" behavior in S-203 may offer the last audience but must never commit it as the default. Any send broader than the poster's pod, or spanning more than one yard, requires an explicit confirmation that names the audience and its member count ("Both yards, 47 people"). Media posts default to pod regardless of history. Every rendered post shows its own audience, so a mistake is discovered in minutes rather than weeks. Only the bridging household can trigger the multi-yard case, so the friction lands exactly where the risk is.

Answers: T-YARD-1, media sticky-audience gap, T-MINOR-5, sticky-memory-escalates-kid-photos gap. Binds: S-203, S-302, S-202.

### TM-4. Reply-by-email addresses are capabilities, and From: is never trusted

A reply address is a bearer credential exactly like a token. It carries at least 128 bits of entropy in the local part, is unique per member and per post, and is revocable through TM-1. Attribution comes from the address alone. The From: header is only a secondary consistency check: a mismatch with the member's registered address quarantines and surfaces on the admin delivery-status panel, and never attributes. Bounce responses are byte-identical for "no such thread" and "not your thread" so the bounce channel is not an existence oracle. This holds for S-502 now; when S-503 ships it inherits the same bar and never regresses to From:-only.

Answers: T-EMAIL-1, T-EMAIL-2, T-TOKEN-8, reply-address WRITE gap (yard), reply-spoofing gap (minors). Binds: S-502, S-503.

### TM-5. Token hygiene: capability ceiling, URL-to-cookie exchange, no-referrer, hashed at rest and in logs

The elder token grants read, one-tap react, and email-mediated comment only. No profile edit, no directory contact fields, no member-list browsing, no export, no invite. A leaked token leaks the feed, never the address book or the instance. On first open the tokenized URL is exchanged for an httpOnly session cookie and redirected to a clean URL, so history, referrers, and screenshots afterward show no token, and the original link still works on any device with no re-auth. Referrer-Policy is set by the app per surface so a token never rides a cross-origin Referer: `no-referrer` on the form-less token-in-URL surfaces (`/t/`, `/d/`, `/media/`), and `same-origin` on the surfaces that must host a CSRF-protected form (`/e/`, `/join`, `/break-glass`, `/digest` confirm/unsubscribe, and the admin hand-over pages). `same-origin` still suppresses every cross-origin Referer — so a URL token never reaches a third party — while sending the same-origin `Origin` that Django's CSRF check requires; a blanket `no-referrer` makes browsers send `Origin: null` on a same-origin form POST, which CSRF rejects, silently breaking every form. The app owns this (`SECURE_REFERRER_POLICY` floor + `TokenSurfaceHeadersMiddleware`/`serve_media`); the edge must NOT override it. Tokens are at least 128-bit CSPRNG values, stored only as a hash (a leaked hobbyist backup does not leak live links), and appear in app and proxy logs only as a short hash prefix. All token, digest-link, and signed-media routes carry X-Robots-Tag noindex and a robots.txt disallow. Every bearer credential in the system meets the entropy and hash-at-rest bar, and invalid tokens return the same 404 as an unknown route.

Answers: T-TOKEN-1, T-TOKEN-3, T-TOKEN-4, crawler-indexing gap, elder-token-leak-in-use gap, T-EDGE-1, T-REPO-2. Binds: S-102, S-104, S-403, S-301.

### TM-6. A deceased-member state, distinct from removal

Death is near-certain over a multi-year instance: the founding family has four grandparents among roughly sixty people, and elders are exactly the token holders. Families will not run "removal" on a grandmother who died, because it reads as erasing her and forces a keep/anonymize/delete decision while they grieve. So the design provides a separate "mark as passed" action. It fires the TM-1 revocation handler (token, sessions, digest, reply addresses all die at once), stops the digest, and freezes the profile out of the birthday and anniversary surfaces, while keeping content and attribution intact. No cheerful "Nana reacted" event, no birthday banner for someone the family buried. One line in the admin runbook names the action so it gets used.

Answers: death-of-token-holder gap (token), death gap (yard, media, minors, auth, operator). Binds: new deceased-member story, S-102, S-702, S-501, S-903.

### TM-7. Instance lifecycle: handover, shutdown, domain lapse, encrypted backups, non-resurrecting restore

We treat single-maintainer abandonment as the likeliest way this instance dies: self-hosted family software mostly dies of neglect, not attack. That is a judgment call, not a measurement. The principles already plan for it with export. The threat model adds the security half. Backups encrypt by default, with a passphrase set in the first-run wizard and a printable recovery sheet (a lost passphrase is a real availability risk, so the sheet is not optional). Restore ends with a forced security-replay: it rotates the token-signing material and surfaces a checklist of removals and revocations that postdate the backup, so a restore never resurrects a removed ex-partner's account or a revoked token. A documented handover runbook rotates every secret, regenerates all tokens, and transfers registrar and host accounts. A documented shutdown runbook delivers a final export to at least two members and destroys volumes and provider snapshots. The domain is treated as a family asset: auto-renew, multi-year registration, registrar lock, and days-remaining reported in the admin health email, because a lapsed domain hands every printed QR and bookmarked link to a squatter. Docs state plainly that the instance admin can read everything, and that isolation is a member-level promise, not an admin-level one.

Answers: shutdown/handover/domain-lapse gaps (token, yard, minors, auth, operator), admin-plane-bypasses-isolation gap, restore-resurrects gap, colocated-backup gap, bus-factor gap, shadow-email-archive gap, T-BACKUP-1, T-MEDIA-5. Binds: S-704, S-801, S-802, S-803, new handover/shutdown story.

### TM-8. Compose and first-run safety

The shipped compose publishes exactly one port, the reverse proxy. Database, cache, and queue live on an internal network with no host mapping. There are zero default credentials: the first-run wizard creates the admin interactively and generates every internal secret randomly at first boot. The wizard is gated by a one-time setup secret printed only to the server console, and it hard-disables the moment an admin exists, including after a restore or an upgrade that briefly reopens setup mode. The app refuses to boot on an empty or placeholder secret. HTTPS is the zero-config default of the fifteen-minute path through a bundled auto-TLS proxy, and the app refuses to mint token links for a non-HTTPS base URL in production.

Answers: T-COMPOSE-1, T-BOOT-1, T-REPO-1, T-EDGE-1. Binds: S-801, S-902.

### TM-9. Media derivatives inherit the original's access checks

Every media byte, original or derived, is served through the same access-checked endpoint and inherits the pod/yard scope of its source. No static-served derivative directory exists. Thumbnails, resizes, transcodes, and poster frames get independently unguessable identifiers, never derivable from the source ID. Deleting media deletes every derivative in the same job, and outstanding signed URLs are invalidated at the serving layer (deletion beats expiry). Location and identity metadata is stripped server-side at ingest for every entry path, and the strip covers video container atoms (QuickTime/MP4 location), not only photo EXIF, running before any derivative is generated so derivatives are born clean. Uploads are re-encoded at ingest and served with an exact Content-Type, X-Content-Type-Options: nosniff, and Content-Disposition, so an uploaded SVG or HTML polyglot never executes as active content in the app origin.

Answers: T-MEDIA-4, T-MEDIA-3, SVG/active-content gap, video-metadata gap. Binds: S-401, S-402, S-403, S-302, S-202.

### TM-10. Supervised children never get token links, and parental control is overt

Nothing in the token stories restricts who may hold a token, and the token is the lowest-friction path, so a parent will reach for it instead of a supervised account unless the design stops them. Token links cannot be issued to supervised members; the S-104 generation UI names the surface as the elder path and routes children to supervised accounts. Supervision itself is never covert. Supervised status is visible to pod-mates, not only admins, so cousins with full accounts know the account is parent-readable. Supervised surfaces carry no "private" framing and tell the teen plainly that the managing parent can see everything. The single carve-out from quiet-by-default is a non-push notice to the managing parent when their supervised child joins or creates a pod, so Backyard is never the least visible channel in the family. The product's promise is no platform surveillance, never protection from one's own parents, and the docs say exactly that.

Answers: token-links-for-kids gaps (token, minors), elder-token-to-child gap, T-MINOR-2, T-MINOR-3. Binds: S-102, S-104, S-701, S-703, S-305.

## 3. Per-boundary threats

Every kept threat below has a row. Boundary-specific gaps that no TM absorbed are folded in with a `-G` id. Threats whose whole fix is a TM point at it rather than restating it. Likelihoods reflect the adversarial pass's adjustments.

### 3.1 Token links

The no-login elder token surface, and everywhere a token or a derivative credential travels: digest deep links, signed media URLs, per-post reply addresses, QR printouts, browser history, referrers, server logs, and the revocation lifecycle. The core framing: the token is the credential and the URL is the session, so every copy of the URL is a copy of the access. The design makes copies cheap to observe and cheap to kill, because making them hard to create (expiry, re-auth) fails the elder principle of months of daily use with no passwords.

### 3.2 Yard isolation

The two sides of the family must never see or infer each other's posts, comments, media, profiles, directory entries, members, or existence. The founding six-person household is the only bridging pod; every other member belongs to exactly one yard. Cross-yard access returns 404, not 403, enforced server-side. The interesting failures are not exploits, they are defaults and parallel code paths: a sticky picker, an object-fetch endpoint that trusts an ID, a batch job that reimplements audience logic.

|---|---|---|---|---|

Anything already viewed or emailed cannot be recalled. Bounded to the bridging pod. |

Photos and videos of children across their whole lifecycle: ingest, storage and serving with access checks, derived assets, embedding in the digest and elder surfaces, backups and exports, and deletion. The boundary is crossed whenever a media byte leaves the access-checked path: into an email, a mail-provider image proxy, a browser cache, a backup on a NAS, or a URL someone can copy.

| ID | Threat | Likelihood | v1 answer | Residual |

Children as subjects and users. Supervised accounts, the grooming surface through quiet ad-hoc pods, excluded adults retaining access to children's media, relatives posting kids' photos against a parent's wishes, and honest legal scoping for a non-commercial family operator. Children's media is the instance's most sensitive asset class.

| T-MINOR-6 | A child's school line, address, and birthday in the directory let an in-instance adversary locate the child in the real world. | Medium | Minors' profile fields default to pod-only (S-902); supervised profiles parent-managed; GPS stripped (S-401); birthday visibility extended to per-field with a pod-only supervised default; docs discourage a wider-than-pod school field. | Relatives learn where kids go to school in real life; feed text is uncontrolled. Targets low-effort silent harvest, not omniscience. |
| T-MINOR-7 | A teen exfiltrates family content, including other kids' photos, to public platforms. | Medium | Zero-build design constraint: no external-share button, no public post URLs, no copy-link that resolves without membership (S-403, S-202); who-reacted-not-counts removes the virality scoreboard; a resharing line in the norms docs. | A screen can always be photographed. Watermarking and screenshot detection are refused: screenshot detection does not work on the open web, and forwarding forensics is family-poison. |

Post-v1: family-configurable parent-confirm on supervised pod joins, a private takedown-request lever, and a remembrance flow designed with the deceased-member state (TM-6). Legal scoping stays exactly one honest docs paragraph: child-safe defaults shipped, zero compliance claims, operator responsibility acknowledged, consult counsel beyond a private family context. It must never grow into age-verification or consent machinery.

### 3.5 Email

Inbound and outbound mail: digest generation and delivery over BYO SMTP, reply-by-email comment posting, bounces and DSNs, unsubscribe, and the signed media and token links embedded in digest bodies. Email is the elder lifeline and the leakiest channel in the product: unrecallable, forwardable by habit, and retained by the relay.

| ID | Threat | Likelihood | v1 answer | Residual |
|---|---|---|---|---|
| T-EMAIL-1 | From: spoofing posts a comment as a family member into the highest-trust space, in threads with children. | High | TM-4 capability address is the credential, From: only a consistency check that quarantines on mismatch, via-email badge, RCPT-time rejection of unknown recipients. | An attacker with both a leaked address and a matching spoofed From: posts to one thread. Badged, revocable, one post's blast radius. |
| T-EMAIL-2 | A forwarded digest leaks reply capabilities, so a stranger or excluded relative posts as the elder. | Medium | TM-4 plus reply addresses expire per digest issue and are revoked on removal (TM-1); docs state that forwarding the digest forwards the ability to reply as you. | Within the validity window a forwarded-digest recipient can forge comments on that digest's posts. | T-EMAIL-3 | A forwarded or compromised mailbox becomes a full read capability because digest links open without a login wall. | Medium | TM-2 and TM-5 per-digest expiring read-scoped tokens, never the master token; short-expiry media URLs; a "Nana's email was hacked" recovery runbook; ADR-003. | Inline photos and text in the email body are held by whoever has the email. | T-EMAIL-8 | Outbound injection: a CRLF-laden kinship name or a crafted title breaks digest headers or injects HTML, phishing elders under the family's own sender. | Low | Strip control characters from all user fields at the input boundary; user content reaches headers and HTML only through a real mail library and the template engine's encoding; extend the digest test with injection fixtures. | Near zero with a real mail library. Post-v1: an anomaly quarantine for replies from unregistered infrastructure, a per-member "disable reply-by-email" toggle, and a linked-instead-of-inline image option. S-503 email-photos-in inherits TM-4 and TM-9 when it ships.

### 3.6 Auth and sessions

Accounts, invites, sessions, recovery, and removal: every path that creates, extends, or destroys a live credential. Elder tokens and digest links are adjacent boundaries, pulled in here wherever a lifecycle event must revoke them. The instance is on the public internet by necessity (elder links and digests work from anywhere) and the repo is public, so every route is attacker-known.

| ID | Threat | Likelihood | v1 answer | Residual |
|---|---|---|---|---|
| T-BOOT-1 | An unclaimed first-run wizard lets a stranger become instance admin; a botched restore can reopen setup mode with data present. | Low | TM-8 one-time setup secret printed to the console; wizard hard-disabled whenever an admin exists, including after restore and upgrade. | Effectively none once the setup token ships. The mitigation is nearly free, which is why it is v1 despite low likelihood. |
| T-INVITE-1 | A forwarded household invite grants pod and yard membership to an outsider who signs up in under two minutes with no identity signal. | Medium | Invites expire and carry a use cap, revocable; joining is an explicit POST; admin sees who joined from each invite; one-action removal (see T-YARD-G1). | Within the expiry window a forwarded invite works. Accepted: the joiner stands out in a small roster and removal is clean. ADR-003. |
| T-CRED-1 | Credential stuffing or an ex who knows a reused password lands a password-fallback account. | Medium | Per-account and per-IP rate limiting with backoff; uniform auth errors (enumeration leaks family structure); reject top-common passwords at set time; a member with a passkey can drop the password entirely. | A targeted adversary who knows the exact password gets one clean login before limits engage. Indistinguishable from the victim at the protocol level. |
| T-SESS-1 | Long-lived stateless tokens make any single session theft permanent, and S-702's "revoke sessions immediately" becomes unimplementable. | Medium | Opaque random tokens with server-side records, individually revocable; httpOnly/Secure/SameSite cookies, rotation on login and privilege change; sliding expiry with an absolute cap; sign-out-everywhere; removal deletes session rows. | Between theft and the member noticing, the thief has full access, and by calm design noticing is slow. The cap and revocability bound it. |
| T-SESS-2 | Shared or unlocked family devices: whoever holds the tablet is that member, and on an admin device can mint tokens or export the instance. | High | Fresh re-auth for sensitive actions regardless of session age: credential/email changes, invite creation, token generation, removal, role changes, export. | Casual reading and posting from an unlocked household device remains. The software boundary is the household, not the individual, unless a member opts into a lock. |
| T-RECOV-1 | Admin-mediated recovery (many members have no email) is a first-class impersonation vector via social engineering. | Medium | A one-time expiring reset link the admin never sees the result of; reset revokes all sessions (fraud locks out the real owner, the detection signal); the affected member is notified on next sign-in; recovery-etiquette docs. | An admin who is the adversary or is thoroughly fooled defeats this. Collapses into T-ADMIN-1; the recovery root of trust is a human. |
| T-ADMIN-1 | Instance-admin takeover is total archive compromise: one login is a remote equivalent of server shell over all yards and one-click export. | Medium | Admin roles require passkey or TOTP, enforced in the wizard so a password-only admin never exists; fresh re-auth for export, role changes, token generation; append-only export and role event log; honest "choose your admin like the keeper of the photo albums" docs. | A malicious admin or an attacker with server shell is beyond the app. Counterweights: honesty, every member's export right, small-family accountability. |
| T-REMOVE-1 | Removal that misses a channel: the removed ex keeps the digest, reply addresses, live media URLs, or a still-valid invite they created. | High | TM-1 atomic revocation over the full channel inventory with a completeness test; short signed-URL expiry; the inventory lives in the S-701 permission matrix so new channels must register. | Everything already delivered, downloaded, or exported is theirs forever. The goal is to stop the flow of new family life, completely and testably. |
| T-REMOVE-2 | Supervised-account credentials survive a parental split: the removed parent logs in as the child. | Low | TM-1 custody step: removal detects supervised accounts the member manages, forces credential rotation, and reassigns to the remaining guardian. | While both parents are members, both can act through the child's account. A governance matter; the docs say whoever manages a supervised account can act as it. |
| T-AUTH-G1 | Mandatory admin 2FA (from T-ADMIN-1) creates lockout risk with no defined recovery; a lazy web recovery flow becomes the very front door 2FA closed. | Medium | Break-glass admin recovery requires server shell by construction: a one-time console command prints a short-lived reset URL, never a web flow (same trust anchor as the setup secret). Docs recommend a second admin and document succession. | The sole-admin-dies case still needs a named second admin or documented succession; the console path is the technical half. |
| T-AUTH-G2 | A yard-A admin runs recovery or token-minting for a bridging member, then logs in and reads yard B, a surface their role can never see. | Medium | Recovery, token minting, and removal for any member whose yard set exceeds the acting admin's scope require the instance admin (v1: only the instance admin initiates recovery); the member-visible reset notice stays. | A single-instance-admin family concentrates this trust in one person, which is disclosed. Retrofitting scope checks after code exists is where it gets expensive. |
| T-AUTH-G3 | A removed ex re-enters through a still-live invite created by someone else (the original household invite from their onboarding). | Medium | TM-1 removal lists all live invites scoped to the removed member's pods and yards and revokes them by default; the S-701 inventory registers "invites reaching the member's pods" as a removal channel. | An invite used in the gap before the removal completes. Bounded by invite expiry either way. |

Content-level moderation (S-713 single-item takedown) is scoped by VISIBILITY, deliberately not by the subset-scope of the person-level lever above (T-AUTH-G2 / can_manage_member): a moderator may take down a post or comment they can see (the ratified "scoped to visible" rule), enforced by routing the item through the same read guard. So an admin who belongs to both yards through a bridge household may moderate content in either, and a both-side post is one object a single-side admin removes from both feeds. This diverges from T-AUTH-G2's subset check by design — the takedown grants no new sight (it is bounded by, not a source of, visibility), is silent, and never touches a member's credentials — so it is not a cross-yard escalation of the kind T-AUTH-G2 addresses. A stricter audience-subset rule (a yard admin may take down only content whose whole audience is within their yards) is a possible post-v1 refinement, flagged for the founder's manual QA.

Post-v1: a per-member named-device list, a "new device signed in" calm banner, optional TOTP for ordinary members, and a designated per-member recovery contact. Device-fingerprint step-up is refused as theater for this scale.

The public repo, the compose file, the edge, backups, upgrades, monitoring, and the operator themselves. The operator is a family member subject to the same ruptures the other boundaries model, and we treat abandonment as the likeliest end-state of any instance, so lifecycle work carries as much weight here as any active attack.

### 3.8 Spec contradictions to reconcile now

The review passes found places where two stories, read together, already contradict each other or fail a promise silently. These are cheap to fix while the stories are still text, and expensive to unwind after code. They are captured as amendments in story-deltas.json; recorded here so the editor sees the reasoning.

- S-102 says "configurable expiry," while the token framing promises months with no re-auth. The default must be no-expiry, or R3 fails silently for every family that never opens settings. Record the default in ADR-003.
- S-402 (video) says nothing about metadata, but MP4 and QuickTime files carry location in container atoms that photo-only EXIF stripping does not touch. S-402 needs a strip line mirroring S-401, or an estranged co-parent reads home and school coordinates off a downloaded clip.
- S-903 makes birthday month and day a required profile field auto-broadcast to every yard member's mailbox, while S-902's per-field visibility covers contact fields only. Combined with a school line and an address, this completes a locate-the-child package. Extend per-field visibility to birthdays with a pod-only default for supervised profiles.
- S-203 says both "defaulting to the pod" and "remembering last choice." The remembered broad choice is the misdirection landmine. The last choice may be offered but must never be the committed default (TM-3).
- S-403's short signed-URL expiry (needed for revocation) breaks S-501's late-opened digests unless TTLs are tiered by surface. The generation-id check, not the TTL, is the revocation mechanism; the TTL only caps mailbox archaeology.
- S-501's "links open without a login wall" and S-502's reply addresses turn any forwarded digest or read mailbox into a read-and-write credential. Both must be scoped, expiring, and treated as capabilities (TM-4, TM-5), and the residual belongs in ADR-003.

## 4. Design tensions we accept

Calls where the product and the security posture pull against each other. Each is one line of tension and one line of where we land.

1. Elder token longevity versus leak surface. R3 demands months of daily use with no re-auth, so expiry and login walls are off the table. We land on long-lived but easy to kill: safety comes from the capability ceiling, named attribution, and instant regeneration, never from making Nana authenticate.
2. A digest that "just works" versus email retention and forwarding. No-login links and inline images are the elder experience; both make the digest a forwardable, retained bearer artifact. Elder usability wins, and we bound the blast radius with per-digest scoped tokens, low-resolution renditions, and expiry, all disclosed in ADR-003.
3. Reply-by-email reciprocity versus address expiry and From-spoofing. Elders reply to three-week-old digests, so a hard-expired address breaks the loop, and From: is trivially forged. We choose a generous window (valid until superseded plus a grace period) and lean on capability addresses, via-email badges, and revocation instead of short expiry.
4. Sticky audience memory versus silent scope-widening. S-203 remembers the last choice for convenience; a remembered broad choice is the trap named in 3.8. We offer the last choice but never commit it as the default, and confirm on any widen.
5. Who-reacted reciprocity versus cross-yard identity leak. Showing "who reacted" is the reciprocity the product is built on, but on a bridging post it leaks the other yard's members. We filter reactors per the viewer's yard, weakening the signal slightly on bridging posts to hold the boundary.
6. People-dates visibility versus contact-field privacy. Birthdays and anniversaries are surfaced yard-wide by design, while phone and address are per-field scoped. We make that an explicit documented decision, with a pod-only default for supervised profiles and year and age never shown.
7. Parent oversight versus teen privacy expectation. A parent-managed account cannot be both parent-controlled and private. We choose honesty over pretense: no "private" framing on supervised surfaces, and the TM-10 promise applied here, that Backyard runs no platform surveillance and never pretends a supervised account is hidden from the parent who manages it.
8. Calm and no-surveillance versus security detection and audit. Every security event wants to notify someone, and the product forbids push and per-person activity feeds. Security signals ride only the calm surfaces (the admin health email, a next-visit banner, a pull-based device list), and the audit log is scoped to auth and admin events, member-visible, and never an input to any metric. Detection is structurally slower, on purpose.
9. Self-hosted admin omniscience versus the isolation promise. The app cannot stop the person who owns the database, and E2EE is a non-goal because it breaks the elder surfaces. The counterweights are non-technical: honest docs, mandatory admin 2FA, and first-class export as every member's exit.
10. Encrypted backups versus lost-passphrase archive loss. Default encryption protects the archive from a stolen NAS but a lost key destroys it, which violates "migrations never eat the archive." We land on default-encrypt plus a wizard-printed recovery sheet plus an exercised decryption drill; a loud --no-encrypt exists for those who accept the risk.
11. Fast, boring, public-repo posture versus secure-by-default and patch cadence. The fifteen-minute bar and the public repo fight hardening, and pinning fights timely patching. Security is the default path (the wizard generates secrets, the proxy does auto-TLS), a separate hardening appendix is refused, upgrades are deliberate and prompted rather than auto-pulled, and security never leans on obscurity.
12. Rooms, not referees, versus consent over children's photos. The product ships levers (a narrow default, delete, house rules, a later takedown-request) and refuses referee machinery (no face or subject tagging of children, no content scanning). The residual conflict is governed by the family, and the docs say so.

Two smaller calls recorded rather than expanded: GPS is stripped at ingest and discarded, trading archive fidelity for safety, and any future retained-original store must sit inside this same model; a departing member's own authored content stays exportable while their access to the yard's feed ends, and a departing pod's past posts get an explicit humane keep/anonymize/delete choice.

## 5. Out of scope, on purpose

Threats the adversarial pass pruned, with the pruner's reasoning kept:

- Timing and existence oracle on cross-yard 404s (was T-YARD-3). A micro-timing side channel against a hobbyist VPS to learn an object count is nation-state-shaped effort for near-zero yield, and an authenticated relative already knows the bridge exists. The one real fix (byte-identical 404 bodies) is folded into T-YARD-2.
- Referer and copy-image-address media leak (was T-MEDIA-7). Modern browsers strip cross-origin paths by default, so the Referer vector is nearly dead (older webviews and embedded clients aside), and copy-image is a trusted member re-sharing one photo, below the software's line. The two free lines (no-referrer, short feed-URL expiry) are folded into TM-5 and T-MEDIA-1.
- Unsubscribe abuse (was T-EMAIL-7). Its whole mitigation is "implement S-501 as written," and the worst case is denial-of-digest for one member, admin-visible and reversible. Two implementation notes carried into S-501 instead.
- Supply-chain fleet blast radius (was T-SUPPLY-1). Enterprise theater when the fleet is one family and the residual admits no hobbyist will verify a signature or an SBOM. The real hygiene (digest pinning, maintainer 2FA, lockfiles) is folded into T-UPGRADE-1 and T-REPO-1. Revisit in Phase 2 or when deployed families grow past a handful.

Deliberately refused, not deferred: watermarking and screenshot detection, face or subject tagging of children, device fingerprinting and login-anomaly detection, and per-person activity logs. Each is surveillance-shaped, does not work, or fights S-705.

Out of scope by product decision:

- End-to-end encryption. A non-goal; it breaks the elder surfaces. The reasoning lives in the PR-FAQ, and the v1 threat model is platform-elimination, not cryptographic secrecy from the operator.
- A malicious instance admin, beyond honest documentation of their power. Self-hosting puts the database in the admin's hands; the control is choosing the right human and every member's export right, not software.
- Nation-states and targeted professional attackers. See section 1.

## 6. How this binds

This model is enforced through story acceptance criteria and the tests they become in the Phase 3 story loop, not through good intentions. The proof lives in these acceptance tests:

- S-202: the isolation suite. Cross-yard 404 for feed, members, media, search, single-object fetches of every type, autocomplete and reactor lists, admin endpoints, and digest rendering.
- S-702: revocation completeness. After removal, leave, and pod-leaves-yard, every credential class returns 404 or bounces on the next request.
- S-403: media access checks. Every asset, original and derived, access-checked; unguessable ids; re-encode at ingest; no active content.
- S-501 and S-502: the digest runs through the same authorization engine as the feed, per recipient, against live state, and reply addresses are capabilities with From: never trusted.
- S-801: compose and first-run defaults. One port, no default credentials, setup secret, HTTPS, admin 2FA.

The proposed story amendments and the three new stories these requirements demand are in story-deltas.json alongside this document. ADR-003 records the token-link forwarding trade-off (and, by extension, the digest-forwarding and inbox-perimeter residuals) separately, as the README already promised. ADR-002 chose the stack, and the stack-specific pass that followed it is section 7, whose rows bind to the same story acceptance criteria this section describes.

## 7. Stack-specific threats (Phase 2)

ADR-002 chose Django 5.2 + htmx + Alpine, PostgreSQL 18, Procrastinate, a four-container compose (caddy, web, worker, postgres), django-allauth, Anymail, Pillow, and ffmpeg. This section is the pass that choice earned. Sections 1 to 6 hold under any stack; these threats exist because of what the stack does by default, and they divide cleanly:

- **present-config** threats are checkable in the repository today. The scaffold is a first-admin walking skeleton, so many present-config rows are latent: correct now, dangerous the moment a later wave adds the surface (a token route, an upload endpoint) that makes them bite. Each says which.
- **planned-surface** threats bind to a library or feature a future wave installs (allauth, htmx, Pillow, ffmpeg, Anymail). They are recorded now so the wave that builds the surface builds the defense with it, as a [wave-plan](../wave-plan.md) acceptance test, not a retrofit.

The same rule as sections 2 and 3 holds: a threat whose whole answer is an acceptance test names that test, and the test is where the proof lives. Nothing here is built. The adversary calibration is section 1's, unchanged: no nation-states, no enterprise supply-chain theater. Keeping section 5's precedent of showing the pruner's reasoning: this pass pruned two classes of finding and does not list them, a handful that only restated a design-level threat already in sections 2 or 3, and a handful that were enterprise-shaped for a one-family box (cosign or SBOM supply-chain verification, a WAF, mTLS between compose containers), the same posture section 5 took toward supply-chain theater.

Two findings surfaced from several boundaries at once and are stated once here, because both are single points that several waves depend on:

- **The proxy hides the client IP (TS-EDGE-IP).** In the four-container topology only Caddy is meant to connect to gunicorn, so Django's `REMOTE_ADDR` is the Caddy container's address. Every per-IP control the model commits to (T-CRED-1, T-EDGE-2 rate limiting) will either see one shared bucket for the whole family or no per-attacker signal at all. Worse, django-allauth's 2026-02-13 security release (65.14.2) makes its built-in rate limits distrust `X-Forwarded-For` by default unless the deployment declares its proxy hop count. v1 answer, armed the wave allauth installs: set a trusted-proxy-count of exactly one (the bundled Caddy, which overwrites client-sent forwarded headers) and derive the client IP only from the value Caddy wrote, never `REMOTE_ADDR`; verify against the library docs at the build wave both the exact setting name (`ALLAUTH_TRUSTED_PROXY_COUNT` as of 65.14.2) and that 65.14.2 flips `X-Forwarded-For` trust off by default. The mitigation (trust exactly one proxy) is correct whichever way that verification lands. The one-trusted-hop assumption is only true once the network split of TS-CO-4 lands: on today's flat compose network a sibling container can reach `web:8000` directly and forge a one-hop-legitimate `X-Forwarded-For`, so this control and TS-CA-3's spoofed-header test depend on that split. Acceptance test drives requests through the full compose stack and asserts two IPs get two buckets across worker processes (see TS-DJ-13). Residual: correct only while the one-hop topology holds. A CDN or a second proxy in front silently changes the count and mis-keys again, and IP-keyed limits are inherently weak against a distributed sprayer, so per-account backoff carries the real weight. This is the same invariant that `SECURE_PROXY_SSL_HEADER` already leans on (settings.py lines 129 to 132). Binds: T-CRED-1, T-EDGE-2, S-101.
- **The logs are armed to print token URLs from two sources, not one (TS-EDGE-LOG).** `entrypoint.sh` line 28 runs gunicorn with `--access-logfile -`, and gunicorn's default format logs the full request line including path and query string. Dropping that flag is necessary but not sufficient, because Django's own `django.request` logger writes `Not Found: /t/<token>` at WARNING to stderr on every 404, and the token surface guarantees 404s: expired tokens, revoked tokens, and TM-5's own 404-parity probes all take that path, so a stale-link click writes a live-shaped bearer token into the logs even after the gunicorn flag is gone. No token routes exist yet, so nothing leaks today; the moment S-102 lands, both sources are live. v1 answer: drop `--access-logfile -` from the entrypoint (Caddy is the edge and logs access under TS-CA-2's redaction rule), AND add a `LOGGING` config with a redaction filter on `django.request` and `django.security` (and the same for `DisallowedHost` and 500-path records) so no request-path token reaches stderr. The TM-5 acceptance test greps every container log and must exercise the 404 and invalid-token paths, not a 200: a grep written against a valid-token request false-greens forever. Residual: redaction is only as good as its filter, so the grep is a standing test across all log sources, re-run as routes change. Binds: TM-5, T-REPO-2, T-TOKEN-3, S-102.

### 7.1 Django application layer

The framework's defaults are mostly safe; the danger is the safe default that is load-bearing but unstated (a later "simplify" change flips it), and the escape hatch that turns one templating mistake into a same-origin write. Acceptance: a connection that sends no headers is closed within about ten seconds. | Read-header timeout is the slowloris control; the slow-body worker-pinning is bounded by `read_body` and buffering, and a distributed attack against a hobbyist box is out of scope for section 1's adversary. |
| TS-CA-7 | [present-config] The Caddy admin API is correctly off (`admin off`), but re-enabling it for config reloads is the natural next step a maintainer takes, and an exposed admin API on `localhost:2019` inside the container is a config-rewrite primitive. | Low | Keep `admin off` with a drift-guard grep; reloads go through a container restart, not a live admin socket. | If a future need forces the admin API on, it stays bound to loopback and off the published surface; the grep makes the change visible. |

Binds: TM-5, TM-8, T-EDGE-1, T-EDGE-2, T-REPO-2, S-102, S-401, S-402, S-801.

### 7.4 Compose and containers

The compose file is the product's install experience and its blast-radius boundary, and TM-8's promises (one port, internal DB network, no default credentials) are partly comments rather than config, with cheap container hardening left on the table. Files audited: docker-compose.yml, Dockerfile, entrypoint.sh, Makefile, .env.example.

| ID | Threat | Likelihood | v1 answer | Residual |
|---|---|---|---|---|
| TS-CO-1 | [present-config] The default `json-file` Docker log driver never rotates. On a small VPS this is a disk-fill vector and permanent plaintext retention of whatever the app logs (see TS-EDGE-LOG, TS-PG-4). | Medium | Set `logging` with `max-size` and `max-file` on each service (or a daemon default documented in deploy docs); pairs with the health email's disk-headroom field. | Rotation bounds retention, it does not redact; the log-content rules (TS-EDGE-LOG, TS-PG-4) do that. |
| TS-CO-2 | [present-config] The entrypoint auto-migrates on every boot with no pre-flight backup (entrypoint.sh line 20), directly against T-UPGRADE-1's committed "auto-migrations with a mandatory pre-flight backup". | Medium | Before `migrate`, dump to `/data/backups` with the version-matched client (TS-PG-6), keep the last few, and refuse to migrate if the dump fails; an explicit env override skips it (S-803). Acceptance: a boot with a failing dump does not run migrations. | An operator who force-sets the override owns the outcome; the pre-flight backup is the backstop, not a guarantee against a determined skip. |
| TS-CO-3 | [present-config] The web entrypoint hardcodes gunicorn and runs `migrate`/`collectstatic`/`ensure_setup` unconditionally, with no `CMD` passthrough. ADR-002's worker container is the same image running `procrastinate worker`, so as written the worker either never starts jobs or races the web container's migrations. | Medium | Make the entrypoint role-aware before the worker wave: end it with `exec "$@"`, add a gunicorn `CMD` to the Dockerfile, and gate migrate/collectstatic/ensure_setup on the web role so the worker waits and does not re-migrate. | A worker that boots before web migrates crash-loops until web is up; `depends_on` health ordering plus the role gate contains it. |
| TS-CO-4 | [present-config] The compose declares no `networks`, so TM-8's "DB on an internal network with no host mapping" is a comment, not config, and every container shares one default network. That makes web reachable from the worker (the SSRF-sensitive link-preview container), which is what breaks TS-CA-3's and TS-EDGE-IP's one-trusted-hop invariant, and makes postgres reachable from Caddy. | Medium | Three networks, not one shared internal net: `edge` (caddy plus web), `web-db` (web plus postgres), `worker-db` (worker plus postgres). web and worker never share a network because they communicate only through Postgres and Procrastinate, so only Caddy can reach web and only web and worker can reach the database. Acceptance: postgres is unreachable from Caddy's namespace and web is unreachable from the worker's. | Segmentation raises the bar inside the host; a host-level compromise still sees everything, which is the operator boundary (section 3.7), not this control. |
| TS-CO-5 | [present-config] Secrets ride the process environment: `POSTGRES_PASSWORD` and the exported `DJANGO_SECRET_KEY` are visible via `docker inspect` and inherited by every child process (including future ffmpeg subprocesses). The Makefile today sets no umask at all, so `make up` writes `.env` (with the DB password) mode `0644`, world-readable on a shared host. | Medium | Put a `umask 077` inside the same compound command that writes `.env` (so it lands `0600`); keep the DB password in the compose env (the app needs the DSN) but scope what the app role can do (TS-PG-1); ensure ffmpeg subprocesses do not inherit secret env they do not need. Acceptance: `.env` is mode `0600` after `make up`. | Env-var secrets are inherent to compose; the mitigations shrink the surface (file mode, least-privilege role, child-env scoping) rather than removing it. |
| TS-CO-6 | [planned-surface] No memory or pids limits on any container. One crafted upload driving Pillow or ffmpeg to exhaustion can OOM the whole box, taking the family's instance down. The shared web/worker image is fine at this scale; limits are the missing control. | Medium | Set `mem_limit` and `pids_limit` on the worker (and web) sized for the target hardware, so a parser bomb kills its own container, not the box; concurrency-1 on the transcode queue already caps parallelism. Acceptance: a bomb upload OOMs only the worker container. | Limits turn a box-wide outage into a contained one and a failed job; the parser caps (TS-PP-2, TS-PP-3) are the upstream fix. |
| TS-CO-7 | [present-config] Cheap runtime hardening is left on the table: the container runs writable-rootfs with a chowned `/app/staticfiles`, no `read_only`, no `cap_drop`, no `no-new-privileges`, no `tmpfs` for `/tmp`. A future RCE can then persist by rewriting app files. | Low | `read_only: true` rootfs with `tmpfs` for `/tmp`, `cap_drop: [ALL]`, `no-new-privileges: true`; move `collectstatic` to build time with a throwaway ephemeral key (ManifestStaticFilesStorage hashes are content-based, not secret-derived) and drop the runtime chown so the app cannot rewrite itself. | These raise the cost of persistence after an RCE; they do not prevent the RCE, which the parser and template rules address. |
| TS-CO-8 | [present-config] Pinning is asymmetric: compose digest-pins postgres and caddy, but the Dockerfile floats `python:3.13-slim` and the uv image on tags only, so a rebuild can silently pull a changed base against S-803's archive-reproducibility. | Low | Digest-pin the python and uv images too, and treat a base bump as a deliberate change with a migration note; balances against S-803 upgrade discipline (pinning is not never-patching, it is patching on purpose). | Digest pins must be advanced deliberately to get security patches; the discipline is scheduled review, recorded in the upgrade runbook. |

Binds: TM-8, T-UPGRADE-1, T-MON-1, T-REPO-2, S-801, S-802, S-803.

### 7.5 The parsing pipeline

Every finding here is planned-surface: the parsers bind to future waves (media, digest, link previews). They are the instance's contact with genuinely hostile bytes, so they are recorded now to be built with their defenses. Sources: ADR-002's media and email sections, threat-model sections 3.3 and 3.5, TM-9.

| ID | Threat | Likelihood | v1 answer | Residual |
|---|---|---|---|---|
| TS-PP-1 | ffmpeg auto-detects playlist and concat demuxers on uploaded video, so a member upload can turn into a local-file-read or external-fetch primitive through a crafted container or an MP4 external-data reference (`dref`). | Medium | Two distinct controls, because they close different halves: `-protocol_whitelist file` (dropping http/tcp/rtp) closes the external-fetch half but leaves `file` allowed, so it does not stop local reads; the local-read half is closed by `-f` forcing the exact expected container (so playlist and concat demuxers never engage) plus ffmpeg's `enable_drefs` staying off (never pass `-enable_drefs`), which stops a `dref` atom reading `/data/secret_key`. Add `-nostdin`, and run any ffprobe pre-pass with the identical flags. Binds S-402, TM-9. | A decode-time CVE in a whitelisted demuxer remains possible; re-encode-at-ingest and the resource caps (TS-PP-2) are the standing hedges. A later change that drops `-f` while keeping the whitelist silently reopens local file read. |
| TS-PP-2 | ffmpeg transcode has no committed timeout or resource limit, so one pathological clip wedges the family's worker indefinitely. | Medium | Wall-clock timeout and rlimits on the ffmpeg subprocess, on the concurrency-1 queue ADR-002 already commits; a clip that exceeds the timeout fails the job with a clear message (S-402's "never a silent failure"). | TS-PP-3 | Pillow's decompression-bomb default only warns (it does not error) up to twice the pixel limit, so the framework default lets a bomb through the parse before anything rejects it. | Medium | Set `Image.MAX_IMAGE_PIXELS` to error, not warn; enforce a format allowlist at `open`; run the parse worker with warnings-as-errors so a bomb is a rejected upload, not a log line. | TS-PP-5 | The link-preview fetcher (S-301) is a server-side URL fetch, so SSRF is closed only if redirects, DNS rebinding, and IP-literal bypasses are all handled. ### 7.6 The Postgres RLS decision (ADR-004)

ADR-002 carried one question forward: yard isolation (S-202) is enforced in the application layer (membership-scoped managers, `get_object_or_404`, the isolation suite, a mandatory-guard CI rule); should Postgres row-level security sit underneath it as belt and suspenders? A for/against/evidence panel with a judge decided it, recorded in full as [ADR-004](../adr/ADR-004-rls.md). The ruling and its security-relevant residual, in brief:

**Decision: defer RLS, with triggers, and pay now only for the parts that get expensive later.** The panel verified against current Django docs that the application-layer control has a real structural gap: related-object traversal uses `Model._base_manager`, which Django requires not to filter, so a relation walk bypasses the scoped-manager half of the control. The guard-plus-suite control is enumerative and fail-open: it protects the read paths the suite and the lint know about. That argues for RLS. Against it: RLS on this product's audience predicate (a post is visible if its audience includes a yard the viewer shares or the viewer's pod, plus per-field directory visibility) is a partial second implementation of the audience logic, the exact parallel-authz hazard TM-2 closed; the system principals (the Procrastinate worker, the digest builder) need cross-member reads, so they need a bypass path that recreates the unguarded-batch hole; and per-recipient GUC cycling on pooled connections is a documented source of a new stale-context cross-yard leak, a failure the control would manufacture. The only maintained dedicated Django RLS package (`django-rls`) is single-maintainer and silent on connection pooling, and no comparable self-hosted Django product we surveyed (pretix, Zulip, Sentry, PostHog) uses RLS for its own isolation; the named sources are in ADR-004.

The dividing line the ruling drew is cost-structure: everything whose price rises with deployed instances ships now, everything whose price recurs on the maintainer defers. **Build now** (none is in the repository yet; all land at the start of wave 1, listed there, and item one is also a section 7 row):

- The `backyard_migrator`/`backyard_app` role split (TS-PG-1), so the app never runs as superuser and the RLS-can-never-bind topology is fixed for free.
- An indexed denormalized `yard_id` on single-yard tables with a database-level same-yard constraint on cross-row references, which blunts the `_base_manager` traversal gap without a second engine and keeps a coarse RLS policy one migration away. Multi-yard audience tables have no single `yard_id`, so they stay enforced by the audience module and the isolation matrix; ADR-004 names the boundary.
- A single audience-resolution module consumed by the feed, the object guard, search, reactor lists, the digest builder, and the token surface, with the unscoped manager renamed and CI-allowlisted and a no-raw-SQL grep.
- The S-202 matrix generated from the model registry (a new model with no isolation fixture fails the build) and extended with relation-traversal fixtures.

**The residual, stated plainly:** until a trigger fires, yard isolation rides on one enforced perimeter with no database net beneath it. One read path that evades both the scoped managers and the generated S-202 matrix (a `_base_manager` traversal the fixtures missed, a drive-by PR's new route, a management command, a future admin-health aggregate) is a live cross-yard leak, caught by a human noticing rather than a database refusing. The build-now items shrink that surface but do not close it. A secondary residual: the reopening triggers are wave-retro discipline, not CI, so a skipped retro silently converts "defer with trigger" into "defer forever". The triggers, the owner (the wave retro), and the full-kit rule for how RLS ships if adopted are in ADR-004.
