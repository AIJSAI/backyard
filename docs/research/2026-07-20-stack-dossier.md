# Stack dossier (ADR-002 evidence)

Date: 2026-07-20. This is the raw evidence behind [ADR-002](../adr/ADR-002-stack.md): seven candidate stacks, four cross-cutting studies, and a four-lens scoring panel. Every agent in this run was an AI research agent (the same agent-per-candidate method as [ADR-001](../adr/ADR-001-name.md)), not a human evaluator. Load-bearing version and capability claims were verified against live sources on the date above (GitHub releases, npm and PyPI, endoflife.date, and Context7 library docs). Where an agent could not verify a claim it is flagged unverified, and those flags are carried into ADR-002's Phase 2 validation gates.

## Next.js 16 + Drizzle

This stack wins on founder fit, container economy, and ecosystem completeness: strict TypeScript end to end, exactly 3 containers (Caddy, one Node app, Postgres) with no Redis and no object store, and a current, actively-released library for every one of Backyard's unusual needs, including in-process inbound SMTP (smtp-server + mailparser) and a Postgres-native job queue with cron (pg-boss), both verified shipping releases within the last three weeks. The migration story is its quietest strength: drizzle-kit emits plain versioned SQL applied by migrate() at startup, so the S-803 old-DB CI upgrade test is trivial and the family archive is never hostage to the ORM. The honest costs are real: Drizzle's stable line is still 0.x (1.0 at RC, with verified API deprecations coming), Next.js runs an annual-major treadmill with roughly two-year support windows that directly taxes the 'boring to operate' promise, and the mail stack rests on effectively one author. Nothing in the v1 story list is blocked by this stack; the two riskiest stories (S-402 video latency, S-502 inbound email) are hardware- and network-shaped risks that every candidate stack shares, but this one at least keeps them in one language and one process model.

Verified versions (2026-07-20): next.js 16.2.10 (2026-07-01; Next 16 is current major, released Oct 2025); node 24 Active LTS (24.18.0; security support to 2028-04-30); postgresql 18 (18.4; supported to Nov 2030); drizzle-orm 0.45.2 stable (2026-03-27); 1.0.0-rc.4 / 1.0.0-beta.22 on npm dist-tags; caddy v2.11.4 (2026-06-03); better-auth 1.6.23 (2026-06-29), incl. @better-auth/passkey and @better-auth/drizzle-adapter; simplewebauthn 13.3.2 (2026-06-24); nodemailer 9.0.3 (2026-06-30).

Strengths:
- Founder fit is maximal: strict TypeScript across app, jobs, email, and tests, the founder's strongest production skill, and TS/React is the largest OSS contributor pool of any framework for a public AGPL project.
- Genuinely minimal ops surface: 3 containers, no Redis (pg-boss verified: cron + retries + exactly-once on Postgres SKIP LOCKED, in-process), no object store, single app image, few candidate stacks get the digest cron, transcode queue, AND inbound SMTP into one process honestly.
- Every unusual ecosystem need has a CURRENT, actively-released library, verified July 2026 releases for nodemailer (9.0.3), mailparser (3.9.14), smtp-server (3.19.2), pg-boss (12.26.1), sharp (0.35.3), react-email (6.9.0), execa (10.0.0): the mail-in/mail-out story is first-class in Node in a way few ecosystems match.
- Archive-safety story is best-in-class: drizzle-kit emits plain versioned SQL, migrate() applies at startup, the family's data is plain Postgres + files on disk, the S-803 old-DB CI upgrade test is a straightforward job, and principle 7/8 (export, never eat the archive) never depends on the ORM's survival.
- sharp's verified DEFAULTS are S-401's acceptance criteria: strips all metadata including GPS EXIF unless told otherwise, autoOrient() fixes vertical phone photos, the privacy-critical behavior is the zero-config path, not an option you can forget.
- Better Auth's passkey plugin (verified) supports passkey-first registration with no pre-existing session via requireSession:false + resolveUser, an exact mechanical fit for S-101's invite-link-to-feed-in-2-minutes flow, with password fallback and a first-party Drizzle adapter.

Weaknesses:
- Drizzle ORM's stable line is still pre-1.0 (0.45.2) with 1.0 at RC, verified API deprecations between v0 and v1 (.enableRLS() → withRLS()) guarantee one toolchain migration early in the project's life, and the March 2026 release fixed a CWE-89 SQL-injection bug in its identifier-escaping layer: responsive maintenance, but proof the escape hatch had a hole. For a 5-year archive promise, betting the query layer on a pre-1.0 API is the single most legitimate reviewer objection (mitigated, not erased, by plain-SQL migrations).
- Next.js is an upgrade treadmill that fights 'boring to operate': verified annual majors (13→16, Oct 2022–Oct 2025) with ~2-year support windows, a history of App Router caching-semantics churn across majors, and direction set by Vercel, whose commercial center of gravity is its own cloud, not NAS self-hosters. Self-hosting via standalone output is verified and real, but it is the second-class citizen path.
- Bus factors in load-bearing libraries: the entire mail stack (nodemailer, mailparser, smtp-server) is effectively one author's ecosystem (unverified head-count, but long-standing reputation); SimpleWebAuthn is a single maintainer; web-push's last GitHub release is from 2023 (npm 3.6.7 exists but cadence is slow); Serwist is a small project; browser-image-compression is dead (avoided by design). pg-boss is substantially one maintainer too, though verifiably active (release 4 days ago).
- Better Auth is young and moving fast (v1.6.23 with a high release cadence and a passkey plugin whose no-session flow shipped only in the 1.6 line), velocity now, but a 5-year API-stability record does not exist; the SimpleWebAuthn fallback is the insurance.
- No framework-native multi-tenancy or authz: pods/yards isolation, token-link auth, and per-post audiences are all hand-rolled app code, the highest-stakes code in the product is exactly the code no library provides, on a stack whose 2025 middleware-bypass CVE showed how easy layered-auth mistakes are (data-layer enforcement + Postgres RLS + the isolation suite are mandatory, not optional).
- Single-runtime contention: web serving, image processing, and transcode supervision share one Node container on weak hardware; ffmpeg-as-subprocess and sharp's threadpool keep the event loop clear, but a skeptical reviewer will correctly note there is no true isolation until the worker is split into its own container.

Story risks:
- S-402 (short video, minutes-latency): the stack neither helps nor hurts, ffmpeg does the work, but post-to-visible latency on a NAS-class CPU is UNPROVEN and must be benchmarked on target hardware before the caps are documented; software-only H.264 encode of a 60s clip can approach or exceed real-time on weak CPUs, and a transcode running beside the Next.js server on a 4GB NAS risks memory pressure. Mitigation is design (strict caps, 720p preset, pg-boss concurrency 1, reject-over-cap upfront), not stack. Hardware acceleration (QuickSync) is a post-v1 escape hatch.
- S-502 (reply-by-email in): the libraries are first-class (smtp-server + mailparser, same author ecosystem as Nodemailer), but running an inbound MTA on a residential NAS hits port-25 blocking by most consumer ISPs and requires an MX-reachable path (VPS relay, mail-forwarding service, or tunnel), a deployment burden EVERY stack shares, but it must be documented as a first-class setup step or S-502 silently fails for most self-hosters. Spam/abuse filtering on the inbound listener is also entirely yours.
- S-501 (digest deliverability): self-hosted direct SMTP send will land in spam; the compose defaults must require relay credentials (family's own provider) and the docs must own SPF/DKIM setup, again stack-independent but the 'one-command' promise (S-801) is in tension with email requiring external DNS/relay configuration.
- S-202 (multi-tenant 404-not-403): Next.js/Drizzle has NO framework-native multi-tenancy, every query is membership-scoped by convention, and a single missed join leaks existence. Mitigations are real but are all discipline: a single data-access layer that requires a viewer context, Postgres RLS as defense-in-depth (verified supported in Drizzle via pgPolicy/withRLS), and the story's own mandated isolation test suite. Related history: Next.js had a middleware auth-bypass vulnerability in 2025 (CVE-2025-29927, from training data, not re-verified live), the lesson is to enforce authz in the data layer, never only in middleware.
- S-803 (migrations never eat the archive): the archive itself is safe (plain SQL migrations, plain Postgres), but drizzle-kit is pre-1.0 and the verified v0→v1 changes (e.g. .enableRLS() → pgTable.withRLS()) mean the migration TOOLCHAIN will need one supervised upgrade when v1 lands, schedule it before building deep, or start on the RC.
- S-305/S-103 (push on iOS): verified that iOS Web Push works ONLY for Home-Screen-installed web apps, a member who never installs gets no replies-to-me push on iPhone. Acceptable given the tiny opt-in surface, but the opt-in UI must gate on installed state to avoid a broken promise.

Ops: Three containers: Caddy 2.11 (TLS/HTTP, automatic certificates), one Node 24 app container (Next.js 16 standalone server + pg-boss workers + smtp-server inbound listener, all in one process; ffmpeg and sharp's prebuilt libvips baked into the image), and Postgres 18. No Redis (pg-boss uses Postgres), no object store (media on a bind-mounted volume, served through access-checked Next route handlers with signed URLs). Verified: `output: 'standalone'` emits a minimal server.js honoring PORT/HOSTNAME, a single GHCR image, ~200-300MB. Memory: estimated (NOT measured) ~250-400MB for the Node app at family scale, ~150-250MB Postgres tuned small, ~40MB Caddy, comfortable on a 4GB NAS with headroom for sharp spikes; ffmpeg transcodes are the contention point and run concurrency-1. Migrations: drizzle-kit generates versioned plain SQL committed to the repo; the container entrypoint runs a pre-flight pg_dump then drizzle's migrate() before the server binds (S-803's auto-migrate-with-backup verbatim); the CI archive-compatibility test restores a seeded v0.1 dump into a postgres service container and runs the current migrator. Backup (S-802): one documented command, pg_dump + tar of the media volume. Upgrade: docker compose pull && up -d; the entrypoint does the rest. If transcode load ever hurts the feed, the worker splits into a fourth container from the SAME image with a one-line compose change, an escape hatch, not a v1 requirement. Runtime lifecycle on the 5-year horizon: Postgres 18 is supported to Nov 2030 (zero or one DB major upgrade), Node 24 LTS to Apr 2028 (one Node bump), but Next.js majors are annual with ~2-year support, budget roughly one framework major upgrade per year, which is the single biggest recurring ops tax of this stack.

Unverified (carried to Phase 2 gates):
- RAM footprint figures for the app/Postgres/Caddy containers are engineering estimates, not measurements, the S-801 clean-machine run must produce the real numbers.
- ffmpeg transcode wall-clock time for capped clips on NAS-class CPUs, no live source; must be benchmarked on target hardware before S-402's latency promise is committed.
- Whether Better Auth's passkey plugin uses SimpleWebAuthn internally, not confirmed; the two are listed as primary and fallback independently.
- Nodemailer/mailparser/smtp-server being effectively single-author (Andris Reinman), consistent with long-standing public knowledge but commit-stat verification was not performed this session.
- Serwist's maintainer head-count / bus factor, release activity verified (May 2026), team size not.
- CVE-2025-29927 (Next.js middleware auth bypass, March 2025), cited from training data; not re-verified against a live advisory this session.
- Residential ISP port-25 blocking prevalence, well-established ops knowledge, not verified against a live source.
- ffmpeg version shipped in current Debian/Alpine base images, not checked; pin explicitly in the Dockerfile.

## SvelteKit 2 + Drizzle

This stack delivers the best ops-to-capability ratio of the plausible candidates: three containers total (Node app, Postgres, Caddy), with jobs/cron (pg-boss on SKIP LOCKED), outbound email, inbound SMTP parsing, and image processing all running in-process in one TypeScript codebase, no Redis, no Postfix, no separate worker. Its structurally strongest fits are the elder token surface (SvelteKit form actions verifiably work with zero client-side JavaScript) and S-401 (sharp strips GPS-EXIF by default and auto-orients, the story's two trickiest criteria are library defaults). Every load-bearing library was verified alive this week, and the flagship adjacent project (Immich) ships its web UI on exactly SvelteKit 2 + Svelte 5, which blunts the contributor-pool objection. The honest costs are real: the founder writes Next.js/React, not Svelte (a genuine ramp on a new reactivity model that itself was rewritten in 2024); Drizzle stable is still 0.45.x with 1.0 only at RC; SvelteKit 3 is in active development today, guaranteeing a major migration inside the 5-year horizon; and the email-digest templating path is DIY because the Svelte email ecosystem is dead. A very good stack whose biggest risks are founder-fluency and library youth, not capability gaps.

Verified versions (2026-07-20): @sveltejs/kit 2.70.1 (npm, published 2026-07-20); svelte 5.56.6 (npm, 2026-07-16); @sveltejs/adapter-node 5.5.7 (npm, 2026-07-20); drizzle-orm 0.45.2 stable; 1.0.0-rc.4 on rc dist-tag (npm); drizzle-kit 0.31.10 (npm, 2026-06-27); postgresql 18 (18.4 latest minor; EOL 2030-11-14; endoflife.date); node 24 Active LTS (security to 2028-04-30); 26 is Current (endoflife.date); caddy v2.11.4 (GitHub release 2026-06-03).

Strengths:
- Fewest moving parts of any full-featured candidate: 3 containers (app, Postgres 18, Caddy 2.11.4). pg-boss 12.26.1 provides queues AND cron scheduling on plain Postgres (SKIP LOCKED, 'PostgreSQL 13 or higher', no other services), no Redis, no worker container; digest cron, transcode jobs, and push all live in the one Node process.
- Elder surface is a structural fit: SvelteKit form actions are progressively enhanced and work without client-side JavaScript (verified in current kit docs), S-102/S-601/S-602 one-tap reactions can be plain HTML forms on the token route, the most robust possible path for old browsers and webviews.
- S-401's two hardest acceptance criteria are sharp's DEFAULT behavior: 'By default all metadata will be removed' (GPS EXIF strip) and autoOrient() ('Auto-orient based on the EXIF Orientation tag, then remove the tag'), verified in sharp 0.35.3 docs.
- Migrations story matches Principle 8 exactly: drizzle-kit generates plain versioned SQL files; drizzle-orm's migrate() is documented 'safe to run on every application startup because already applied migrations are automatically skipped', S-803's seeded-old-DB CI upgrade test is a straightforward restore+migrate job, and hand-auditable SQL files are the right archive-safety posture for a 5-year horizon.
- Email in AND out stay in-process and in one maintained family: nodemailer 9.0.3 (out), mailparser 3.9.14 (parse), smtp-server 3.19.2 (optional self-hosted inbound listener), all Nodemailer-project packages with July 2026 publishes; inbound reply-by-email needs no Postfix container.
- S-101 passkeys are assembled, not built: Better Auth 1.6.23 ships a passkey plugin (@better-auth/passkey), email/password, a Drizzle adapter, and SvelteKit integration (all verified in current docs); documented fallback is @simplewebauthn/server 13.3.2 (active, June 2026) with hand-rolled sessions.

Weaknesses:
- Founder fluency is the biggest real cost: James is fluent in Next.js/React strict TS, not Svelte. Svelte 5 runes are a different reactivity model, and one Svelte itself replaced wholesale in the 4→5 rewrite (verified migration guide: components are no longer classes, new server API). svelte-check template typing is good but below TSX-grade inference. Weeks of ramp on the project's only engineer.
- Framework churn is live, not historical: sveltejs/kit has an active version-3 branch (last commit 2026-07-20) and a '3.0' milestone with 22 open issues (verified via GitHub API), a SvelteKit 3 major migration WILL land inside the 5-year maintenance window. Svelte's 3→4→5 history shows majors can be paradigm shifts, not just breaking changes.
- Drizzle is pre-1.0 on the stable channel: drizzle-orm latest is 0.45.2 with 1.0.0 only at rc.4 (verified dist-tags), drizzle-kit at 0.31.10, and 1,915 open issues on the repo. Either adopt an RC now or plan a 0.x→1.0 migration in year one; the schema-diffing migration generator is younger and less battle-hardened than Django/ActiveRecord/Prisma equivalents.
- The Svelte email ecosystem is dead where React's thrives: svelte-email last published 0.0.4 in Feb 2023 (verified npm). S-501's inline-image, multipart, old-mail-client digest, a first-class feature, rides on a DIY render()+juice+hand-built-plaintext pipeline with no maintained component library behind it (react-email has no Svelte peer).
- Bus-factor concentration in load-bearing libs: sharp is effectively one maintainer (lovell), pg-boss one (timgit), SimpleWebAuthn one (MasterKale). Better Auth is young (2024-era), fast-churning (already reshuffling plugins into @better-auth/* packages), and its 5-year trajectory is unproven. web-push's npm package hasn't shipped since Jan 2024 (GitHub release v3.6.5, Aug 2023), though the repo shows pushes as recent as 2026-07-20.
- Single-process blast radius: SSR, cron, digest sending, inbound SMTP, sharp, and ffmpeg children share one Node process/container. On a 4GB NAS an ffmpeg or sharp OOM can take the whole app down with it; requires deliberate child-process memory caps, pg-boss concurrency=1 for transcodes, and compose mem_limit settings, disciplined config, not a framework guarantee.

Story risks:
- S-501 (email digest out): highest ecosystem-gap risk on the v1 critical path. No maintained Svelte email library exists (svelte-email dead since 2023); the digest's table-layout HTML, inline images, multipart plaintext fallback, and old-client compatibility are all hand-built on render()+juice+nodemailer. Fully doable, but budget real build-and-test time (Litmus-style manual passes) that a react-email stack would partially amortize.
- S-502 (reply-by-email in): the libraries are strong and in-process (smtp-server + mailparser, both active July 2026), but self-hosting inbound SMTP on a residential NAS is operationally hostile regardless of stack (ISP port-25 blocks, PTR records, spam). The design must ship BOTH the direct smtp-server listener and a provider inbound-webhook path (Postmark/Mailgun/SES) or S-502 fails for most real self-hosters. Risk is ops-shaped, but this stack must own the dual-path complexity.
- S-402 (video, minutes-latency on NAS hardware): ffmpeg spawned from a pg-boss job is the right shape, but actual wall-clock on Celeron/ARM NAS CPUs is UNVERIFIED, no benchmark was run. Meeting 'minutes at most' plausibly requires capping output at 720p/faster-preset and rejecting oversize uploads upfront; a prototype benchmark on target hardware must precede the ADR being marked final. Also: no maintained high-level Node ffmpeg orchestrator, plain child_process is the plan, which is fine but hand-rolled.
- S-305/S-103 (push): web-push 3.6.7 is the least-maintained load-bearing dependency (npm publish Jan 2024). The VAPID protocol is frozen so risk is contained, and the surface is tiny by design, but a skeptical reviewer should note Backyard may end up vendoring or forking it within 5 years. iOS push additionally requires the PWA be installed to the home screen (platform constraint shared by every stack; not re-verified this session).
- S-101 (passkeys): Better Auth is young and moving fast (1.6.23, plugin package reshuffles). If it pivots or breaks, the documented fallback, @simplewebauthn/server 13.3.2 + hand-rolled sessions, must be a real ADR plan-B, because auth is the one component a family instance can never tolerate breaking on upgrade.
- S-202 (pod/yard isolation, 404-not-403): hand-rolled at the repository layer with zero framework scaffolding; the isolation test suite over feed/members/media/search is the ONLY guardrail. This is the stack's largest correctness-risk concentration, an unscoped Drizzle query is silent until a test catches it.

Ops: 3 containers: (1) app, Node 24 LTS (Active LTS, security-supported to 2028-04-30) running the adapter-node build, with pg-boss workers, smtp-server listener, and ffmpeg binary baked into the same image; (2) postgres:18 (18.4 current, community-supported to 2030-11-14); (3) caddy:2.11.4 for auto-TLS/HTTP3/static. No Redis (pg-boss uses Postgres SKIP LOCKED), no Postfix (in-process smtp-server or provider webhook), no separate worker. Estimated idle RAM, NOT measured: Node app 150-300MB, Postgres tuned small 100-250MB, Caddy 30-50MB; under ~600MB idle with transient ffmpeg/sharp spikes to be capped via compose mem_limit, comfortable on a 4GB NAS. Migrations: drizzle-kit-generated versioned SQL applied by migrate() at container start (documented safe-on-every-startup), with an entrypoint pre-flight pg_dump prompt per S-803; the CI archive-compat test restores a seeded v0.1 dump and runs migrate. Upgrades: docker compose pull && docker compose up -d, one app image to rev; Caddy renews certs itself; Postgres major-version upgrades (roughly one per 2-3 years if tracking majors) remain the standard dump/restore or pg_upgrade procedure every stack shares and must be a documented, scripted release-notes step. Backup (S-802): one script, pg_dump + media-volume tar.

Unverified (carried to Phase 2 gates):
- ffmpeg transcode wall-clock times on Celeron/ARM NAS-class CPUs, no benchmark run this session; 'minutes at most' for S-402 is an engineering estimate contingent on a 720p output cap and must be prototyped on target hardware before ADR sign-off.
- RAM footprint figures (Node 150-300MB, Postgres 100-250MB, Caddy 30-50MB) are estimates from experience, not measurements of a built Backyard image.
- iOS Safari requires the PWA to be installed to the home screen for Web Push (iOS 16.4+ platform behavior), widely documented but not re-verified against Apple sources this session.
- Better Auth's governance/funding/bus-factor details beyond npm/docs activity, could not verify team size or sustainability posture live.
- Drizzle's native RLS policy support quality (claimed to exist in recent versions), not verified this session; the S-202 plan does not depend on it (app-layer enforcement + tests is primary).
- fluent-ffmpeg maintenance status, not checked because the recommendation avoids it entirely (plain child_process).
- SvelteKit per-route csr=false page option (useful to keep the elder token route JS-free), well-known docs feature but not explicitly re-verified this session; the elder-surface case rests on the verified no-JS form actions instead.
- Relative OSS contributor-pool sizes (React vs Svelte vs Django), directionally certain, no live quantitative source fetched; the Immich data point is the only grounded evidence offered.

## React Router 7 + Drizzle

This stack is a genuinely strong fit for Backyard v1: two containers (Node app + Postgres), everything in the founder's strongest language (strict TypeScript), and every hard requirement, passkeys, email in/out, images, Postgres-backed jobs with cron, has a currently-maintained library I verified live this session. Its two best properties map directly onto the two binding principles: pg-boss eliminates Redis and Drizzle's plain-SQL migration files plus a startup-safe migrate() make the S-803 "old-DB upgrade in CI" test almost trivial, because the archive lives in Postgres 18 and committed SQL, not in the ORM. The honest weaknesses: React Router now ships a major per year (Remix→RR7 2024→RR8 June 2026), so the 5-year horizon buys ~4 more majors of upgrade tax, softened but not erased by the future-flags path; Drizzle is still 0.45.x with v1.0 only at RC and patched an SQL-injection bug in sql.identifier() in March 2026; and web-push is in barely-maintained mode. The hardest stories (reply-by-email on a residential NAS, minutes-latency transcode on NAS CPU, hand-rolled 404-not-403 tenancy) are hard on every candidate stack, this one at least gives you first-class, recently-released pieces (smtp-server 3.19.2, postal-mime 2.7.5) and a single process to reason about. Verdict: viable and probably the lowest-total-cost option for this founder, provided you start on v8, pin drizzle-kit, and spike S-402 transcode latency on real NAS hardware before committing.

Verified versions (2026-07-20): react-router 8.2.0 (2026-07-08); v8.0.0 released 2026-06-17; v7 still patched (7.18.0 on 2026-06-16); yearly-major cadence; v8 requires Node 22.22+, React 19.2.7+, Vite 7+, ESM-only; node 24 LTS (24.18.0, EOL 2028-04-30); Node 26 becomes LTS 2026-10-28 (EOL 2029-04-30); postgresql 18.4 (major 18 EOL 2030-11-14); drizzle-orm 0.45.2 latest (2026-03-27); 1.0.0-rc.4 on rc dist-tag; @simplewebauthn/server 13.3.2 (2026-06-24); sharp 0.35.3 (2026-07-01); pg-boss 12.26.1 (2026-07-16); nodemailer 9.0.3 (2026-06-30).

Strengths:
- Two-container topology, honestly counted: React Router framework mode is 'just a request handler' on Express (verified in RR docs), so the compose file is app + postgres:18 (+ optional Caddy for TLS). pg-boss keeps jobs and the digest cron in Postgres and in-process, no Redis, no worker container, no cron sidecar. This is the smallest container count any full-featured candidate can hit (S-801).
- Archive-safety story maps 1:1 onto S-803: drizzle-kit generates plain SQL migration files committed to the repo; migrate() from drizzle-orm/node-postgres/migrator runs at container start and 'is safe to run on every application startup because already applied migrations are automatically skipped' (Drizzle docs, verified). The CI old-DB test is: restore seeded v0.1 pg_dump, boot current image, assert. The family archive depends on Postgres 18 (EOL 2030) and SQL files, not on the ORM surviving five years.
- One language, one process, founder's strongest skill: strict TypeScript end-to-end, shared types from Drizzle schema to loader to component. Resource routes (verified: return raw Response instances) cover every non-HTML surface in the same router, signed media URLs (S-403), the inbound-email webhook (S-502), token surfaces (S-102), display URLs, no second service or API layer to keep in sync.
- Every hard ecosystem need resolved to a library with a release in the last ~8 weeks (verified live): SimpleWebAuthn 13.3.2 (Jun 2026), sharp 0.35.3 (Jul 2026), pg-boss 12.26.1 (Jul 2026), nodemailer 9.0.3 (Jun 2026), smtp-server 3.19.2 (Jul 2026), postal-mime 2.7.5 (Jun 2026), react-email 6.9.0 (Jul 2026). The inbound-email pieces in particular are first-class in Node in a way they are not in most ecosystems.
- sharp's defaults ARE the S-401 acceptance criteria: all metadata (including GPS EXIF) stripped by default; autoOrient() applies then removes the Orientation tag so verticals display upright everywhere including email (both verified against sharp docs).
- Governance and contributor pool: React Router moved to an open governance model (remix.run blog, June 2025 post, verified via blog listing), v7 still receives fixes a year after v8 planning began (7.18.0 the day before v8.0.0), and React+TS is the largest OSS contributor pool for a public AGPL project. The upgrade path is institutionalized via future flags: v8's own release notes say adopting v7 future flags makes v8 'a boring upgrade.'

Weaknesses:
- Framework churn is now SCHEDULED, not incidental: Remix → RR7 (Nov 2024) → RR8 (Jun 2026) with an announced yearly major cadence. v8 raised floors hard (Node 22.22+, React 19.2.7+, Vite 7+, ESM-only). Over the 5-year maintenance horizon that is roughly four more majors; future flags make each small, but it is a recurring tax a Django or Rails LTS user would not pay, and 'v1 was built on a framework version that was superseded before the repo's first release' is a bad look the candidate-as-specified (RR7) would invite. Mitigation: start on v8.2.0.
- Drizzle is pre-1.0 six years in: latest stable 0.45.2, v1.0 only at rc.4. The 0.45.2 release itself fixed a possible SQL-injection (CWE-89) in sql.identifier()/sql.as() (verified from release notes), a fair skeptic's exhibit A on maturity. Exposure is bounded because migrations are plain SQL and the query API is what churns, but drizzle-kit output-format drift over 5 years is a real pin-and-vendor discipline requirement.
- web-push is in caretaker mode: last GitHub release Aug 2023, npm at 3.6.7, ~37 open issues. The Web Push protocol itself is frozen-stable which limits actual risk, and S-305's surface is deliberately tiny, but this is the weakest-maintained load-bearing dependency in the list.
- Single-maintainer bus factors on the auth path: SimpleWebAuthn is essentially one person (MasterKale). Active and responsive today (CVE patched within days in June 2026), but passkeys are the primary auth method (S-101) and there is no equally-canonical Node fallback library.
- No framework help for multi-tenancy: the 404-not-403 yard isolation (S-202) is 100% application discipline, RR8 middleware + a membership-scoped repository layer + the acceptance-mandated isolation test suite. Rails (acts_as_tenant) and Django ecosystems have batteries here; this stack hands you primitives and a promise to be careful.
- One process does everything: SSR, sharp, pg-boss workers, and spawned ffmpeg share one container's CPU/RAM. On a NAS this needs deliberate throttles (transcode queue concurrency 1, sharp concurrency cap, --max-old-space-size) or the feed janks during a video upload. Same-image worker-role split via env flag is the escape hatch but adds the third container you were avoiding.

Story risks:
- S-502 (reply-by-email in), the single riskiest v1 story, mostly deployment-topology not library risk: on a residential NAS, inbound SMTP on port 25 is typically ISP-blocked and lacks PTR/reverse-DNS (widely documented; not re-verified this session), so the realistic default is an inbound-parse provider webhook → resource route → postal-mime, which reintroduces a third-party dependency into a 'self-hosted' product and must be documented as such. smtp-server 3.19.2 covers VPS installs. Per-member/per-post reply addresses also require the family's mail domain to route a catch-all, a real hobbyist-operator complexity spike against S-801's ethos.
- S-402 (video, minutes-latency on NAS), UNVERIFIED performance claim: ffmpeg CPU-transcoding a capped clip (e.g. ≤90s, 720p, veryfast) plausibly lands in single-digit minutes on NAS-class x86, but I could not benchmark target hardware this session; this needs a Phase-1 spike before the acceptance criterion is committed. Also fluent-ffmpeg is archived (verified, 2025-05-22), so orchestration is hand-spawned ffmpeg, argv construction, timeout, cleanup, and rejection-over-cap logic are all code you own and test.
- S-202 (404-not-403 isolation), hand-rolled tenancy means one loader that queries without going through the scoped data layer = existence leak. Centralize via RR8 middleware (now always-on) + a repository layer that requires a yard-membership context to construct, and treat the isolation e2e suite as the real enforcement. Postgres RLS as belt-and-braces (SET LOCAL per request) is possible but its ergonomics with Drizzle + a shared node-postgres pool were NOT verified this session.
- S-305 + S-103 (push on iOS), verified platform constraint: iOS web push requires the PWA to be added to the home screen first (iOS 16.4+, WebKit blog). The replies-to-me opt-in must be gated behind install state on iOS or elders/members will hit a dead toggle. Plus the web-push library staleness noted above.
- S-501 (digest: inline images, plain-text fallback, per-member delivery status), rendering and CID inline images are verified-supported (react-email + nodemailer), but 'admin can see per-member delivery status and failures' honestly requires bounce handling, which loops back to the S-502 inbound problem; without inbound, status degrades to SMTP accept/reject plus provider webhook events. Scope the acceptance criterion accordingly.
- S-803 (migrations never eat the archive), drizzle-kit is pre-1.0: pin exact versions, commit generated SQL as the source of truth, and let the CI old-dump upgrade test (cheap on this stack) be the guard against kit behavior drift.

Ops: Containers: 2 required (app: node:24-alpine + ffmpeg + libvips via sharp prebuilt; db: postgres:18-alpine), optional 3rd (Caddy) for TLS if the NAS lacks a reverse proxy, pg-boss runs in-process against Postgres (verified: PG 13+, cron scheduling, no broker), so there is no Redis and no separate worker/cron container; an optional WORKER=1 env-flag role using the same image exists as a pressure valve if transcodes fight SSR. Memory (ESTIMATES, not measured): Node app ~200-400MB idle with SSR + pg-boss, spiking during sharp batches and ffmpeg (ffmpeg runs as a child process so its memory is transient); Postgres 100-300MB with NAS-tuned shared_buffers; comfortably under ~1GB idle on a 4-8GB NAS, with --max-old-space-size set explicitly to avoid OOM ambush. Migrations: drizzle-kit generate at dev time → plain SQL committed → migrate() on container start (verified safe to run every startup, already-applied files skipped), wrapped by an entrypoint that runs a pg_dump pre-flight backup to satisfy S-803's backup prompt; backup/restore (S-802) is one documented command each (pg_dump + tar of the media volume / restore drill in CI). Upgrades: docker compose pull && docker compose up -d, migrations auto-apply; the archive-compat CI job restores a seeded v0.1 dump into postgres:18 and boots the current image from v0.1 onward. Base-layer horizon: Node 24 LTS (EOL 2028-04) now with Node 26 LTS available Oct 2026, Postgres 18 supported to Nov 2030, one Node major bump and zero forced Postgres majors inside the 5-year promise; React Router's yearly major is the one recurring upgrade the maintainer must budget (mitigated by adopting future flags continuously, which v8's release notes describe as making majors 'boring').

Unverified (carried to Phase 2 gates):

- RAM footprint figures (app 200-400MB idle, Postgres 100-300MB), engineering estimates, not measured
- Residential-ISP port-25 inbound blocking and PTR requirements (S-502 topology), widely documented industry behavior but not re-verified live this session
- Postgres RLS ergonomics with Drizzle + a shared node-postgres pool (SET LOCAL per request) as belt-and-braces for S-202, not verified

## Django 5.2 LTS + HTMX

Django + HTMX + Postgres is the closest thing to a purpose-built stack for this spec: the requirements that actually bind, migrations that never eat a 5-year archive (S-803), first-class multipart email in AND out (S-501/S-502), token-link server-rendered elder surfaces (S-102/S-601), and a 4-container compose file a hobbyist can run, map one-to-one onto batteries Django has shipped and stabilized for 20 years, with Postgres as the ONLY stateful service (Procrastinate runs the job queue inside Postgres, so there is no Redis). The honest costs: James trades his strongest language (strict TS) for his second-strongest; the entire media pipeline (client resize, EXIF, ffmpeg transcode, signed URLs) and the PWA layer are hand-assembled from primitives rather than provided; allauth's built-in passkey-signup flow requires mandatory email-verification-by-code, which collides with S-101's "email optional" and forces a custom invite-token signup flow; and htmx is mid-major-transition (2.0.10 stable, 4.0 in beta targeting Summer '26). Nothing in the 33-story v1 cut is impossible on this stack; the concentrated risks are S-402 (video latency is an unbenchmarked estimate on NAS CPUs) and S-502 (inbound email is only turnkey if you accept an ESP inbound route, the pure-NAS path leans on stale or hand-rolled receivers). For a solo typed-Python-fluent maintainer optimizing "boring to operate" over a 5+ year horizon, this is the strongest candidate I can defend with evidence, provided the v1 plan budgets real engineering for media and treats the passkey signup flow as custom code from day one.

Verified versions (2026-07-20): django 6.0.7 current; 5.2 LTS = 5.2.16 (extended support to April 2028); 6.2 LTS lands April 2027 (support to April 2030), recommend pinning 5.2 LTS now, one LTS→LTS hop in 2027; postgresql 18 (18.4 current minor); ~annual majors, 5 years support each (17 EOL Nov 2029, 18 EOL Nov 2030); htmx 2.0.10 stable; v4.0.0-beta5 published 2026-06-26, htmx.org states v4 'in beta, with a target release date of Summer 26'; alpine.js 3.15.12 (2026-04-30); django-allauth 65.18.0 (2026-05-29), supports Django 4.2–6.0; passkey login via MFA_PASSKEY_LOGIN_ENABLED + MFA_SUPPORTED_TYPES=[..,'webauthn']; passkey signup via MFA_PASSKEY_SIGNUP_ENABLED (requires mandatory email verification by code); procrastinate 3.9.0 (2026-06-20); Postgres-native queue, Django contrib, @app.periodic(cron=...) for schedules; django-anymail 15.0 (2026-04-18); normalized inbound webhooks for SES, Brevo, MailerSend, Mailgun, Mailjet, Mandrill, Postal, Postmark, Resend, SendGrid, SparkPost; pillow 12.3.0 (2026-07-01); ImageOps.exif_transpose applies+removes EXIF orientation.

Strengths:
- Direct hit on the binding constraints: S-803 (migrations never eat the archive) is Django's flagship capability, versioned, reversible, squashable migrations with 20 years of production hardening, and the CI old-DB upgrade test is ~15 lines of workflow YAML around pg_dump + manage.py migrate.
- Smallest honest container footprint of any full-featured candidate: 4 containers (caddy/web/worker/postgres), Postgres as the ONLY stateful service, Procrastinate 3.9.0 (verified) runs queue + cron inside Postgres, eliminating the Redis/broker container every Celery- or BullMQ-shaped stack requires.
- Email in AND out is genuinely first-class, not bolted on: Django core does multipart digest email with CID inline images natively (S-501), and Anymail 15.0 (verified) normalizes INBOUND parsing, envelope_recipient for per-member-per-post reply addresses, attachments, stripped_text, across 11 ESPs including self-hostable Postal (S-502).
- The elder/token surface is the stack's native mode: S-102/S-601/S-602 are server-rendered HTML pages needing zero JS, token auth is django.core.signing plus one middleware; no hydration, no bundle, works on a 2015 tablet browser. Progressive enhancement via HTMX means the entire app degrades gracefully, which IS principle 6.
- Django admin gives the instance operator (S-104 token generation, S-702 member removal, S-705 health views) a free, battle-tested back-office from day one, weeks of UI work no other candidate includes.
- 5-year horizon is contractual, not aspirational: Django LTS releases carry ~3 years of security support each on a published schedule (5.2→April 2028, 6.2→April 2030), governed by a foundation (DSF) with multiple paid fellows, the lowest framework-abandonment risk in web development; Postgres majors get 5 years each.

Weaknesses:
- Founder-fluency tax: James's deepest fluency is strict TypeScript/Next.js; this stack idles that skill and bets on his second language. Django idiom (ORM query semantics, migration internals, allauth's adapter architecture) is a real learning curve, and typing is weaker than strict TS, django-stubs/mypy coverage of the ORM is good but partial (typing-layer detail unverified this session), and templates/HTMX attributes are stringly-typed with no compile-time check across the HTML surface.
- htmx major-version churn in flight (verified): stable is 2.0.10 but v4.0.0-beta5 is published with a stated Summer '26 target, v1 either builds on 2.x and absorbs a major later or chases a beta now; I could NOT verify a documented long-term support window for 2.x post-v4 (only the 1.x-for-IE 'in perpetuity' statement). Mitigation is real but partial: hypermedia coupling is shallow and vendored htmx never breaks silently.
- The media pipeline, the highest-effort part of v1, is assembled, not provided: client resize, EXIF strip/transpose, ffmpeg orchestration, signed URL serving are all hand-built from primitives (Pillow/ffmpeg/signing). Django gives excellent primitives and zero pipeline; a Rails-ActiveStorage-style batteries story does not exist here.
- Key-library bus factors a reviewer will raise: django-allauth is effectively one primary maintainer (pennersr) on the most security-critical dependency (mitigated by enormous install base and steady releases, 65.18.0 May 2026); Procrastinate's recent releases are driven overwhelmingly by one maintainer (medihack); pywebpush sits in the small web-push-libs org (2.3.0 July 2026, active but thin); the pure-self-host inbound fallback django-mailbox looks abandoned (April 2024, Django 5.0 classifiers).
- No row-level security by default: S-202 isolation is app-layer discipline (membership-scoped managers + get_object_or_404), one missed scoped queryset is a cross-yard leak. Postgres RLS as belt-and-suspenders is possible but Django has no native RLS management, so it's custom migration SQL. The story's mandated isolation test suite is the real control.
- Sync-first WSGI model: fine for v1 (calm product, no realtime), but long-running work must stay strictly in the worker, and any future live surface (the post-v1 ambient frame could want SSE) pushes toward ASGI/channels or polling, a fork in the road deferred, not eliminated.

Story risks:
- S-101 (passkey-first signup, email optional): allauth's built-in passkey SIGNUP (MFA_PASSKEY_SIGNUP_ENABLED) hard-requires mandatory email verification by code (verified in docs), incompatible with 'email optional' invite signup. The invite-token flow must be a custom view that creates the account then runs allauth's WebAuthn enrollment; doable and testable (Playwright virtual authenticator) but it is custom code against allauth internals, not configuration. Under-2-minute mobile UX also means fully replacing allauth's default templates.
- S-402 (video clips, minutes-to-visible on NAS hardware): zero framework help, ffmpeg orchestration, caps, progress, and failure UX are all hand-rolled. The 'minutes' latency on a Celeron/N100-class CPU is an UNVERIFIED estimate (veryfast-preset 720p H.264 of a ≤60s clip should land in single-digit minutes, but it MUST be benchmarked on target hardware before the acceptance criterion is committed); a single worker means a queue-depth stall can blow the latency budget unless media gets its own Procrastinate queue.
- S-502 (reply-by-email): first-class ONLY with an ESP inbound route (Anymail webhooks, Postmark/Mailgun/SES) or by self-hosting Postal (heavy for a NAS). The fully-NAS-self-contained inbound path is the stack's weakest link: django-mailbox is stale (April 2024) and aiosmtpd means writing your own receiver; plus inbound MX/DNS setup lands on the hobbyist operator regardless of stack. Recommend: v1 documents ESP-inbound as the supported path and treats pure-self-host inbound as post-v1.
- S-401 (client resize + HEIC): client-side resize is hand-rolled Canvas code (no batteries), and iPhone HEIC uploads likely require the pillow-heif plugin server-side, NOT verified this session; validate before locking acceptance criteria.
- S-103 (installable PWA feel): manifest + service worker are trivial, but making an HTMX multi-page app FEEL app-like in standalone mode (no white-flash navigations, sensible SW caching of the shell) takes deliberate work Django gives no opinion on; iOS's share-sheet install flow is platform friction every stack shares.
- S-301 (link preview cards): no maintained batteries, OpenGraph fetching with SSRF protection (private-IP blocking, timeouts, size caps) is hand-rolled security-sensitive code; the old Python oEmbed libraries are unmaintained (maintenance status unverified), so plan to write ~200 careful lines rather than depend.

Ops: 4 containers, one image: (1) caddy for TLS + reverse proxy, (2) web = gunicorn + Django + WhiteNoise (static files in-process, no nginx sidecar), (3) worker = same image running `procrastinate worker` with ffmpeg installed, (4) postgres:18. No Redis (Procrastinate lives in Postgres), no object store (media on a bind-mounted volume, served through an access-checked Django view; S-403 signed expiring URLs via django.core.signing.TimestampSigner, stdlib-of-Django, no dependency). Collapsible to 3 containers by running the worker as a second process in the app container, but 4 is cleaner and still one `docker compose up` (S-801). Memory: ESTIMATE (unverified) ~0.7–1.2 GB total, gunicorn 2–3 workers ~300–400MB, Procrastinate worker ~150–250MB (spikes during ffmpeg), Postgres 100–300MB tuned small, Caddy ~40MB, comfortable on a 4–8GB NAS. Backups (S-802): one script = pg_dump + tar of the media volume; restore drill is compose-down/restore/up. Migrations (S-803): Django's migration framework is the most battle-tested in any web framework, entrypoint runs `manage.py migrate` on upgrade behind a pre-flight pg_dump; the CI archive-compatibility job (old seeded DB → migrate → smoke) is trivially expressible because migrations are versioned Python files in the repo. Upgrade cadence over 5 years: pin Django 5.2 LTS (supported to April 2028), hop to 6.2 LTS (April 2027, supported to 2030), Django's documented two-release deprecation policy makes LTS→LTS hops routine; Postgres majors are annual with 5-year support, and pg_upgrade or dump/restore is documented in the operator guide. For the family operator the entire story is: `docker compose pull && docker compose up -d`, with a backup prompt in front.

## AdonisJS 6

AdonisJS is the strongest "one language, one process, two containers" candidate for Backyard: a strict-TypeScript MVC framework whose first-party batteries (Lucid migrations, session auth with remember-me, mail with CID inline images and one-click unsubscribe, router-level AND storage-level signed expiring URLs, Japa testing with a Playwright browser suite) map directly onto S-102/S-403/S-501/S-803, and whose Node runtime lets inbound SMTP, the job queue (pg-boss on Postgres, no Redis), and ffmpeg orchestration all live in the same process the founder already writes fluently. The honest caveats are real: the framework is on v7 as of 2026-02-25 (the candidate was framed as v6, that alone demonstrates the churn cadence of roughly one major every two years, and the v5-to-v6 move was a full ESM rewrite), core development is ~91% one person, and the four hardest stories (passkeys, video transcode, inbound email, tenant isolation) get zero framework leverage, they are custom integration code on any stack, but Adonis's batteries cover the easy 70%, not the hard 30%. It comfortably beats a Next.js-only architecture on ops simplicity and beats Rails/Phoenix on founder fluency and OSS-contributor TypeScript reach, at the price of betting a 5-year archive on a small-team framework whose mitigation is that everything underneath (Node, Postgres, Knex, nodemailer, sharp, ffmpeg) is boring and huge.

Verified versions (2026-07-20): @adonisjs/core 7.3.5 (2026-07-08; v7.0.0 stable 2026-02-25, docs at docs.adonisjs.com cover v7); @adonisjs/lucid 22.4.2 (2026-04-05; built on Knex 3.3.0, npm-modified 2026-06-26); @adonisjs/auth 10.1.0 (2026-04-09; guards: session w/ remember-me tokens, access tokens, basic, no built-in WebAuthn); @adonisjs/mail 10.4.0 (2026-07-05; added one-click List-Unsubscribe); @adonisjs/drive 4.0.0 (npm 2026-02-25; FlyDrive wrapper, local-FS signed URLs + file server); pg-boss 12.26.1 (2026-07-16; Postgres 13+, cron scheduling built in, no Redis); sharp 0.35.3 (2026-07-01); @simplewebauthn/server 13.3.2 (2026-06-24, security-fix release).

Strengths:
- Two-container compose (app + Postgres) with queue, cron, inbound SMTP, and transcode workers all inside one Node process, pg-boss (Postgres-only, verified) eliminates Redis, and smtp-server embeds in-process; this is the fewest-moving-parts architecture of any full-featured candidate and directly serves S-801 and 'Boring to operate'.
- Founder-fluency exact match: strict TypeScript end to end, one language for web, jobs, mail, and media pipelines; no context-switching between a TS frontend and a Python/Ruby backend, and the OSS contributor pool for TypeScript dwarfs Ruby/Elixir.
- First-party batteries that hit v1 stories verbatim (all verified against current docs): Lucid migrations with production --force/rollback/dry-run (S-803); session auth with remember-me tokens (S-101 fallback); mail with CID inline images, multipart fallback, and one-click List-Unsubscribe added in v10.4.0 (S-501); router-level signed expiring URLs (digest links) AND Drive-level signed URLs with a file server that returns 404 in production for denied files (S-403's 404-not-403 posture built in).
- Japa testing is unusually well-matched to a story-driven CI spec: functional suite with loginAs and per-test global transactions makes the S-202 isolation matrix cheap; the first-party Playwright browser suite covers acceptance-criteria e2e without bolting on a separate harness.
- Server-rendered Edge pages make the elder surfaces (S-102, S-601) fast, low-JS, and robust on old devices, the calm, no-hydration path is the default here, not a fight against a JS metaframework.
- The framework is current and actively shipping: v7 stable 2026-02-25, patch 7.3.5 on 2026-07-08, docs fully on v7; runtime horizon is covered by Node 24 LTS (2028) and Node 26 LTS (2029).

Weaknesses:
- Framework bus factor is the headline risk: 91% of adonisjs/core top-contributor commits are one person (thetutlage/Harminder Virk, 2,237 vs 74 for #2, verified via GitHub API), the core team is ~3 people, and 19k stars is an order of magnitude below Laravel/Rails/Next. For a project whose own research brief names single-maintainer abandonment as the category's top killer, choosing a framework with the same shape of risk is an irony a reviewer will not miss.
- Churn history: majors in 2021 (v5), 2024 (v6, a full ESM/IoC rewrite widely experienced as a hard migration), and 2026 (v7). The candidate brief itself said 'AdonisJS 6' and the current major is 7, proof of how fast the ground moves. The team's 'small breaking changes surface' for 6→7 is a vendor claim I did not independently verify against migration reports. On a 5-year horizon, plan for v8 and possibly v9.
- The 'batteries-included queues' framing is false: AdonisJS has NO first-party queue. The community options are @rlanz/bull-queue (Redis, violates fewest-containers, last publish 2024) or hand-wiring pg-boss yourself; pg-boss is excellent (verified active) but the integration glue, worker lifecycle, and graceful shutdown are yours to own with no official recipe.
- The four hardest stories (S-101 passkeys, S-402 video, S-502 inbound email, S-202 isolation mechanism) get zero framework leverage, Adonis's batteries de-risk the easy 70% (CRUD, auth session, mail out, migrations) while the differentiating 30% is bespoke on this stack exactly as it would be on Fastify or Next.
- Lucid is a decorator-based Active Record on Knex: materially weaker compile-time type safety than Kysely/Drizzle, and its column decorators sit awkwardly with the strictest TS taste; v7's 'end-to-end type safety' pitch is about routes/RPC, not the ORM. Knex itself is alive (3.3.0, June 2026) but slow-moving maintenance-mode infrastructure.
- Edge templates are a niche skill: OSS drive-by contributors overwhelmingly know React/JSX, not Edge, a real friction for a public repo courting contributors, and the counterweight to the TypeScript-pool argument.

- S-402 (short video, minutes-latency): the single biggest risk on ANY stack, and Adonis contributes nothing, the whole pipeline (VineJS size/duration caps, pg-boss transcode queue, execa-spawned ffmpeg, progress/failure surfacing) is hand-built. Implications:
- v1 default posture: ship the BYO-mailbox adapter, IMAP-poll (60s) a dedicated family-controlled mailbox (Gmail or Fastmail; never Yahoo), and send digests via that same account's port-587 SMTP submission. - Docs honesty requirement: 'BYO-SMTP' must be documented as 'bring your own SUBMISSION relay', with an explicit warning that direct-to-MX sending from a home IP will be rejected or spam-foldered (Google PTR-match requirement + Spamhaus PBL, both grounded). Never ship a default that tries direct MX delivery.
- Design the inbound surface as one narrow internal interface (raw MIME or parsed JSON + HMAC, into a single ingest endpoint/queue) with three adapters: imap-poll (default), provider-webhook (Postmark JSON / Mailgun route / Cloudflare Email Worker shapes, this is also the hosted-mode path), and smtp-sink (advanced, VPS-with-clean-rDNS only, embedded via nodemailer/smtp-server rather than a maddy/DMS container).
